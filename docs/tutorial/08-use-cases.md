# 使用场景

## 什么时候用 Specflow

| 场景 | 为什么用 |
|------|---------|
| 新功能开发 | 需求需要澄清，实现需要拆分，完成需要验收——specflow 全流程覆盖 |
| 大规模重构 | 改动跨多文件，需要上下文注入和影响面追踪 |
| 团队协作项目 | spec 库团队共享，编码规范自动注入，新人不用口口相传 |
| 复杂 bug 修复 | break-loop skill 做根因分析，避免反复修同一个问题 |
| 跨会话续作 | journal 记录进度，mem 检索历史对话，新窗口不丢失上下文 |
| 多 agent 并行开发 | worktree 管理支持多个 agent 在独立工作树上并行 |

## 什么时候不用 Specflow

- **简单问答**：问 API 用法、概念解释，直接回答，不建任务
- **inline 小任务**：改几行代码、小修小补，当前回合可完成，直接动手
- **探索性原型**：还在试方向，需求随时变，建任务反而是负担
- **一次性脚本**：写完即弃，不需要验收和沉淀

!!! tip "Consent Gate"
    specflow 有一个"建任务确认门禁"：AI 判断需要建任务时，**不会直接创建**，而是先用一两句话说明原因，取得你同意后才执行 `specflow task create`。简单对话和小改动不会触发任务流程。
