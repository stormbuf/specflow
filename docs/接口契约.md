# Specflow 接口契约

> 本文档定义 specflow 五个模块的接口契约，作为编码前的最后准备。所有结构基于已确认的《重构设计思路》决策，并参考 Trellis 源码的精确数据结构。

---

## 一、specflow-cli/（Go CLI）

### 1.1 全局约定

- **二进制名称**：`specflow`
- **配置目录**：目标项目根下的 `.specflow/`
- **退出码**：`0` 成功；`1` 一般错误；`2` 配置校验失败；`3` session 独占冲突
- **输出格式**：默认人类可读文本；`--json` 输出 JSON（供插件 exec 解析）
- **日志**：`--verbose` 输出详细日志到 stderr；默认只输出结果到 stdout

### 1.2 task.json schema

```json
{
  "id": "2026-06-27-add-breadcrumb-hook-1",
  "title": "为 inject-workflow-state 实现面包屑注入",
  "description": "实现 workflow.md 解析与面包屑注入，规避 #367",
  "status": "planning",
  "intent": "用户反馈每轮都要手动告诉 AI 当前阶段，需要面包屑自动注入",
  "creator": "stormbuf",
  "assignee": "stormbuf",
  "createdAt": "2026-06-27",
  "completedAt": null,
  "vcs": "jj",
  "baseRev": "kkmuqwop",
  "children": [],
  "parent": null,
  "relatedFiles": [],
  "meta": {}
}
```

**字段定义**：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `id` | string | change-id | 任务唯一标识，格式 `YYYY-MM-DD-short-slug-N` |
| `title` | string | 必填 | 任务标题 |
| `description` | string | `""` | 任务描述 |
| `status` | string | `"planning"` | `planning` / `in_progress` / `completed` |
| `intent` | string | `""` | 变更意图（简陋几句话），合并自原 Proposal 阶段，不参与行为规约同步 |
| `creator` | string | `.developer` 值 | 创建者 |
| `assignee` | string | `.developer` 值 | 指派者 |
| `createdAt` | string | 当天 `YYYY-MM-DD` | 创建日期 |
| `completedAt` | string\|null | `null` | 完成日期（archive 时写入） |
| `vcs` | string | `config.yaml:vcs` | 版本管理工具 `git` / `jj` |
| `baseRev` | string\|null | `null` | 任务开始时的 VCS 基线版本（`git rev-parse HEAD` / `jj log -r @- --no-graph -T commit_id`） |
| `children` | []string | `[]` | 子任务 id 列表 |
| `parent` | string\|null | `null` | 父任务 id（子任务时写入） |
| `relatedFiles` | []string | `[]` | 相关文件路径（可选，AI 维护） |
| `meta` | object | `{}` | 扩展元数据（自定义键值对） |

**与 Trellis 的差异**：

| Trellis 字段 | specflow 处置 | 原因 |
|-------------|-------------|------|
| `branch` / `worktree_path` / `commit` / `pr_url` | 移除 | specflow 不管理分支与 PR，由用户 VCS 工作流负责 |
| `base_branch` | 改为 `baseRev` | 记录基线版本而非分支名，适配 jj（jj 无传统分支概念） |
| `dev_type` / `scope` / `package` / `priority` | 移除 | 模板瘦身决策，不引入分类体系 |
| `subtasks` | 移除（与 `children` 重复） | 仅保留 `children` |
| `notes` | 移除 | 由 prd.md 和 journal 承载 |
| — | 新增 `intent` | 保留意图段，合并自原 Proposal |
| — | 新增 `vcs` | 版本管理中立，每个任务记录使用的 VCS |

### 1.3 session 指针结构

**文件位置**：`.specflow/.runtime/sessions/<session-key>.json`（gitignored）

```json
{
  "platform": "opencode",
  "session_id": "abc123",
  "current_task": ".specflow/changes/2026-06-27-add-breadcrumb-hook-1",
  "last_seen_at": "2026-06-27T10:30:00Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `platform` | string | 平台标识 `opencode` / `pi` / `session` |
| `session_id` | string | 平台会话 ID（从 hook input.sessionID 获取） |
| `current_task` | string\|null | 当前活跃任务目录（repo 相对路径），`null` 表示无活跃任务 |
| `last_seen_at` | string | ISO8601 UTC 时间戳，用于 stale 检测 |

**session-key 生成规则**：`<platform>_<session_id>`，其中 session_id 经 sanitize（`[^A-Za-z0-9._-]` → `_`，截断 160 字符）。

**独占检查逻辑**（`task start` 时）：

```
1. 遍历 .runtime/sessions/*.json
2. 若某 session 的 current_task == 目标任务路径 且 session-key != 当前 session
   → 返回退出码 3，输出 "任务 X 正被 session Y 占用，请先 finish 或 task release"
3. 否则写入当前 session 指针
```

**stale 检测**（`doctor` 时）：`last_seen_at` 超过 24 小时未更新的 session 指针标记为 stale，提示清理。

### 1.4 .fingerprints.json 结构

**文件位置**：`.specflow/.fingerprints.json`（gitignored）

```json
{
  "specflow_version": "0.1.0",
  "files": {
    ".specflow/workflow.md": "sha256:abc123def456...",
    ".opencode/plugins/inject-workflow-state.js": "sha256:789abc012def...",
    ".opencode/skills/specflow-brainstorm/SKILL.md": "sha256:3456789abc...",
    ".opencode/agents/specflow-implement.md": "sha256:def456789abc..."
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `specflow_version` | string | 创建/更新此指纹文件时的 CLI 版本 |
| `files` | map<string,string> | key = 文件相对路径，value = `sha256:` + hex digest |

**update 三路比对逻辑**：

```
对每个管理文件 file：
  old_hash = fingerprints.files[file]     // 上次 install/update 时记录
  cur_hash = sha256(当前磁盘文件内容)
  new_hash = sha256(CLI 内嵌新版本内容)

  if cur_hash == old_hash:
      // 用户未修改 → 安全覆盖
      覆盖为新版本
  elif new_hash == old_hash:
      // 用户修改了但 CLI 版本没变 → 保留用户版本
      跳过
  else:
      // 用户修改了且 CLI 也更新了 → 冲突
      交互询问：覆盖 / 合并（写入 .update-candidates/）/ 放弃

update 完成后刷新 fingerprints.files 中所有已覆盖文件的 hash
```

### 1.5 子命令接口定义

#### specflow init

```
specflow init -u <developer> [--opencode|--pi] [--vcs git|jj] [--force]

输入：
  -u           开发者身份（必填）
  --opencode   目标平台 OpenCode
  --pi         目标平台 Pi（预留）
  --vcs        覆盖 VCS 自动检测结果
  --force      覆盖已存在的 .specflow/ 目录

输出（stdout）：
  ✅ specflow 已初始化
  平台: opencode
  VCS: jj (自动检测)
  开发者: stormbuf
  安装位置:
    - .specflow/ (运行时)
    - .opencode/skills/ (7 skills)
    - .opencode/plugins/ (3 plugins)
    - .opencode/agents/ (3 native agents)
  文件指纹已记录

退出码：0 成功 | 1 已存在且未 --force | 2 VCS 检测失败
```

#### specflow task create

```
specflow task create [--parent <dir>] [--title <title>] [--description <desc>]

输入：
  --parent <dir>   父任务目录路径（建子任务时指定）
  --title <title>  任务标题
  --description    任务描述

行为：
  1. 生成 change-id: YYYY-MM-DD-short-slug-N（N 自增避免同日冲突）
  2. 创建 .specflow/changes/<change-id>/
  3. 写 task.json (status=planning)
  4. seed prd.md / implement.md / implement.jsonl / check.jsonl（带 _example 行）
  5. 若 --parent：写子任务 task.json.parent，父任务 task.json.children 追加
  6. 记录 baseRev（git rev-parse HEAD / jj log -r @- --no-graph -T commit_id）
  7. 设当前 session 指针指向新任务

输出（--json）：
  {
    "task_id": "2026-06-27-add-breadcrumb-hook-1",
    "task_dir": ".specflow/changes/2026-06-27-add-breadcrumb-hook-1",
    "status": "planning",
    "parent": null
  }
```

#### specflow task start

```
specflow task start [<task-dir>]

行为：
  1. 解析目标任务：参数指定 > session 指针 > 报错
  2. 检查 session 独占：遍历所有 session 指针，若其他 session 已指向同一任务 → 退出码 3
  3. status: planning → in_progress
  4. 写当前 session 指针
  5. 记录 baseRev（若尚未记录）

输出（--json）：
  { "task_id": "...", "status": "in_progress", "session_exclusive": true }

退出码：0 成功 | 3 session 独占冲突
```

#### specflow task finish

```
specflow task finish

行为：
  清除当前 session 指针的 current_task（设为 null），释放任务独占。
  不改 task.json.status。

输出：✅ session 指针已清除，任务独占已释放
```

#### specflow task release

```
specflow task release <task-id>

行为：
  强制清除指向该任务的所有 session 指针（用于 stale 清理）。

输出：✅ 已释放 N 个 stale session 指针
```

#### specflow task archive

```
specflow task archive [<task-dir>]

行为：
  1. status → completed，写 completedAt
  2. 移动目录到 .specflow/changes/archive/<YYYY-MM>/
  3. VCS auto-commit（按 config.yaml:vcs 分派）
  4. 清当前 session 指针

输出：✅ 任务已归档到 archive/2026-06/
```

## 二、platforms/opencode/plugins/（JS 插件）

### 2.1 OpenCode 插件 API 约定

OpenCode 插件是工厂函数，接收 `{ directory, client }` 上下文，返回 hook 事件处理对象：

```js
export default async ({ directory, client }) => {
  return {
    "hook.event.name": async (input, output) => { ... }
  }
}
```

- `directory`：项目根目录绝对路径
- `input`：事件输入对象（含 `sessionID`、`agent` 等）
- `output`：事件输出对象（可原地修改）
- hook 通过**原地修改 output** 生效，不能替换 output 引用

### 2.2 inject-workflow-state.js

**hook 事件**：`chat.message`（每轮用户消息触发）

**职责**：解析 workflow.md，按当前任务 status 匹配面包屑标签块，非破坏性注入。

**接口签名**：

```js
export default async ({ directory }) => {
  return {
    "chat.message": async (input, output) => {
      // 1. 跳过条件检测
      if (isSpecflowSubagent(input)) return           // 跳过子 agent 消息
      if (process.env.SPECFLOW_HOOKS === "0") return  // 环境变量禁用
      if (!isSpecflowProject(directory)) return        // 非 specflow 项目

      // 2. exec 调用 Go CLI 获取当前任务状态
      const result = await exec(["specflow", "task", "current", "--json"])
      const task = JSON.parse(result.stdout)

      // 3. 解析 workflow.md 面包屑标签块
      const templates = loadBreadcrumbs(directory)
      const status = task ? task.status : "no_task"
      const breadcrumb = buildBreadcrumb(task, status, templates)

      // 4. 非破坏性注入（规避 #367）
      injectNonDestructive(output, breadcrumb)
    }
  }
}
```

**面包屑解析**（`loadBreadcrumbs`）：

```js
const TAG_RE = /\[workflow-state:([A-Za-z0-9_-]+)\]\s*\n([\s\S]*?)\n\s*\[\/workflow-state:\1\]/g

function loadBreadcrumbs(directory) {
  const workflowPath = join(directory, ".specflow", "workflow.md")
  // 读取并解析所有 [workflow-state:STATUS]...[/workflow-state:STATUS] 块
  // 返回 { status: body } 字典
  // 找不到匹配块时不报错，buildBreadcrumb 降级为固定文案
}
```

**面包屑构造**（`buildBreadcrumb`）：

```js
function buildBreadcrumb(task, status, templates) {
  let body = templates[status]
  if (body === undefined) {
    body = "Refer to workflow.md for current step."
  }
  let header = task === null
    ? `Status: ${status}`
    : `Task: ${task.task_id} (${status})`
  return `<workflow-state>\n${header}\n${body}\n</workflow-state>`
}
```

**非破坏性注入**（`injectNonDestructive`）—— #367 规避核心：

```js
function injectNonDestructive(output, breadcrumb) {
  const parts = output?.parts || []

  // 方案 A（优先）：若 OpenCode 支持独立 context part
  // 插入独立 part，不修改用户 message text part
  parts.unshift({
    type: "text",
```

### 2.3 inject-subagent-context.js

**hook 事件**：`tool.execute.before`（每次 task 工具调用前触发）

**职责**：拦截 subagent 派发，exec 调用 `specflow build-context` 取上下文，注入 prompt。

**接口签名**：

```js
export default async ({ directory }) => {
  const config = loadAgentsConfig(directory)  // 读 .specflow/agents.yaml

  return {
    "tool.execute.before": async (input, output) => {
      if (process.env.SPECFLOW_HOOKS === "0") return
      if (input.tool !== "task") return

      const args = output.args
      if (!args) return

      const subagentType = args.subagent_type       // 原名，如 specflow-implement
      const agentConf = config.agents[subagentType]  // 直接查配置，不做前缀替换
      if (!agentConf) return                          // 配置未声明则不处理（扩展点）

      // exec 调用 Go CLI 构建上下文
      const result = await exec([
        "specflow", "build-context", subagentType, "--json"
      ])
      if (result.exitCode !== 0) return

      const context = result.stdout
      const originalPrompt = args.prompt || ""

      // 拼装最终 prompt（上下文 + 约束 + 原始 prompt）
      args.prompt = wrapPrompt(subagentType, originalPrompt, context, agentConf.constraints)
    }
  }
}
```

**wrapPrompt 拼装格式**：

```js
function wrapPrompt(agentType, originalPrompt, context, constraints) {
  const constraintBlock = constraints && constraints.length > 0
    ? `\n\n## 行为约束（必须遵守）\n${constraints.map(c => `- ${c}`).join("\n")}`
    : ""

  return `<!-- specflow-hook-injected -->
# ${agentType} Agent Task

## 上下文

${context}
${constraintBlock}

---

## 你的任务

${originalPrompt}`
}
```

### 2.4 session-start.js

**hook 事件**：`chat.message`（每会话首条消息触发，去重）

**职责**：exec 调用 `specflow get-context`，注入 session context。

**接口签名**：

```js
export default async ({ directory, client }) => {
  const processed = new Set()  // 内存去重

  return {
    "chat.message": async (input, output) => {
      if (isSpecflowSubagent(input)) return
      if (process.env.SPECFLOW_HOOKS === "0") return
      if (!isSpecflowProject(directory)) return

      const sessionID = input.sessionID
      if (processed.has(sessionID)) return  // 本会话已注入

      // exec 调用 Go CLI 获取 session context
      const result = await exec(["specflow", "get-context", "--json"])
      if (result.exitCode !== 0) return

      const ctx = JSON.parse(result.stdout)
      const contextBlock = buildSessionContextBlock(ctx)

      // 非破坏性注入
      const parts = output?.parts || []
      parts.unshift({
        type: "text",
        text: contextBlock,
        metadata: { specflow: { sessionStart: true } }
      })

      processed.add(sessionID)
    }
  }
}
```

**buildSessionContextBlock 输出格式**：

```
<current-state>

活跃任务: 2026-06-27-add-breadcrumb-hook-1 (in_progress) — 为 inject-workflow-state 实现面包屑注入
</current-state>

<spec-indexes>

- .specflow/spec/backend/index.md

</spec-indexes>
```

```yaml
# session 独占配置

  stale_threshold_hours: 24         # stale 指针检测阈值
```

### 3.2 agents.yaml schema

```yaml
# agent 声明
agents:
  <agent-name>:
    source: native | platform | custom   # agent 定义来源
    jsonl_file: <filename> | null        # 任务目录下的 jsonl 文件名
    agent_file: <path>                   # 仅 custom：.specflow/agents/ 下的定义文件相对路径
    require_task: true | false           # 是否要求活跃任务才能运行
    readonly: true | false               # 是否只读（不修改源码）
    can_write: true | false              # 是否可写源码
    constraints:                         # 行为约束列表，注入 prompt
      - "约束文案 1"
      - "约束文案 2"
```

**校验规则**：
- `source: native` → 不需要 `agent_file`
- `source: platform` → 不需要 `agent_file`
- `source: custom` → `agent_file` 必填
- `jsonl_file: null` 表示该 agent 不注入文件上下文（仅注入 constraints）

### 3.3 workflow.md 结构

```markdown
## 阶段索引
- Phase 1: 规划  (status=planning)
- Phase 2: 执行  (status=in_progress)
- Phase 3: 收尾  (status=completed)

[workflow-state:no_task]
无活跃任务。先分类请求：简单对话 / inline 小任务 / 完整 specflow 任务。
建任务需取得用户确认（task creation consent gate）。
[/workflow-state:no_task]

[workflow-state:planning]
流程：brainstorm → 编写 prd.md → 编写 implement.md → (可选) design.md → curate jsonl → task start
加载 `specflow-brainstorm` 进行需求探索。
[/workflow-state:planning]

[workflow-state:in_progress]
流程：implement sub-agent → check sub-agent → 加载 `specflow-sync-requirements` → spec update → commit → finish-work
Dispatch prompt 以 `Active task: <path>` 开头。
上下文：按 jsonl manifest 条目顺序加载。
[/workflow-state:in_progress]

[workflow-state:completed]
任务已完成。运行 /specflow:finish-work 归档任务并写 journal。
[/workflow-state:completed]

### 活跃任务路由
- 规划或需求不清 -> `specflow-brainstorm`
- in_progress 实现/验收 -> 派发 implement / check sub-agent
- 行为规约同步 -> `specflow-sync-requirements`
- 重复调试 -> `specflow-break-loop`；spec 更新 -> `specflow-update-spec`
```

**面包屑标签块语法**：

```
[workflow-state:<STATUS>]
<面包屑文案，中文表述>
[/workflow-state:<STATUS>]
```

- `<STATUS>` 仅允许 `[A-Za-z0-9_-]+`
- 插件用正则 `/\[workflow-state:([A-Za-z0-9_-]+)\]\s*\n([\s\S]*?)\n\s*\[\/workflow-state:\1\]/g` 解析
- 找不到匹配 status 的块时降级为 `Refer to workflow.md for current step.`
- 改面包屑 = 改 markdown，下一条消息即生效

### 3.5 jsonl manifest 格式

**文件位置**：`.specflow/changes/<change-id>/<agent>.jsonl`（如 `implement.jsonl`）

**每行格式**（JSON Lines）：

```json
{"file": ".specflow/spec/backend/coding-style.md", "reason": "后端编码规范"}
{"file": ".specflow/changes/2026-06-27-add-hook-1/prd.md", "reason": "需求文档"}
{"file": ".specflow/changes/2026-06-27-add-hook-1/implement.md", "reason": "执行计划"}
```

**特殊行**：

```json
{"_example": "Fill with {\"file\": \"<path>\", \"reason\": \"<why>\"}. Put spec/research files only — no code paths. Delete this line once real entries are added."}
```

- `_example` 行在 `task create` 时 seed，AI curate 时删除
- 无 `file` 字段的行被 `build-context` 跳过

**支持的条目类型**：

| 类型 | 格式 | 行为 |
|------|------|------|
| 文件 | `{"file": "<path>", "reason": "..."}` | 读取文件内容注入上下文 |
| 目录 | `{"file": "<dir>/", "type": "directory", "reason": "..."}` | 读取目录下所有 .md 文件 |

**路径规则**：`file` 为项目根目录的相对路径。
