# Specflow 重构设计思路

> 本文档基于对 Trellis 源码（`../Trellis`）、Trellis 官方文档（`https://docs.trytrellis.app/zh`）、Trellis issue #367 以及本地设计思路笔记（`Trellis 自定义工作流/设计思路.md`）的全面调研，结合已确认的架构决策与修订意见，提出 specflow 的重构方案。文档覆盖背景、目的、思路、可能的方案、理由与影响面。除技术术语外，正文使用中文表述。

---

## 一、背景

### 1.1 specflow 现状

specflow 当前是一个**纯文档驱动的 8 阶段变更生命周期 skill**，无运行时状态机、无代码、无 CLI。核心结构如下：

| 组成 | 位置 | 说明 |
|------|------|------|
| skill 入口 | `specflow-workflow/SKILL.md` | 阶段路由、审查循环、执行纪律、12 条约束 |
| 阶段定义 | `specflow-workflow/references/*.md`（8 个） | roadmap / proposal / spec-delta / system-architecture-adr / design / tasks / apply / archive |
| 文档模板 | `specflow-workflow/assets/*.md`（10 个） | 共约 1335 行，`design.md` 758 行、`tasks-template.md` 212 行为最重 |
| 安装器 | `install-specflow-workflow` skill | 扫描 `~/project/` 交互式复制到目标项目 `.agents/skills/` |

阶段推进靠三件事：**文件存在性门禁**（产物文件是否存在）+ **slash 命令路由**（命令参数决定进入哪个阶段）+ `tasks.md` 复选框（Apply 内任务级进度）。没有中央状态机文件，没有每轮状态注入。

跨会话「记忆」完全由目标项目内的持久文档承担（`specflow/specs/`、`specflow/architecture.md`、`specflow/adr/`、`specflow/roadmap.md`、`specflow/archive/`），skill 内无 memory / lessons / conventions 子系统。`specflow/state.json` 被 `.gitignore` 忽略且 skill 从不引用它。

### 1.2 四个痛点（带证据）

**痛点 1：缺少自动化 subagent 上下文注入机制，全靠 orchestrator 自觉。**

全仓检索 `inject|subagent|context injection` 零命中。编排器向子 agent 传递上下文靠「文档中介 + prompt 约定」：审查 agent 的 prompt 须包含产物路径/阶段名/审查重点/上轮问题（`SKILL.md` 第 194-202 行规定）；执行类 agent 靠 `tasks-template.md` 内嵌的 Agent 角色标注 + `covers:` 字段 + 编排器读入工作记忆后按需委托（`references/apply.md` 第 25-34 行）。**没有独立注入管道**，子 agent 拿到什么上下文完全取决于编排器是否「自觉」读取并拼装。

**痛点 2：没有 workflow.md 和面包屑机制，上手心智负担重。**

全仓检索 `workflow-state|breadcrumb` 零命中。AI 在每一轮对话中无法自动得知「当前在哪个阶段、下一步该做什么」，只能靠 slash 命令显式路由或人工判断文件存在性。阶段间的进入条件、阻断条件、审查循环分散在 8 个 reference 文件和 `SKILL.md` 中，新人或新会话需要通读才能建立全局心智模型。

**痛点 3：各阶段文档模板过重，每次改动耗时特别长。**

10 个模板共约 1335 行，其中：

| 模板 | 行数 | 问题 |
|------|------|------|
| `design.md` | 758 | 功能族 §1-§8、优化族 §1-§5、迁移族 §1-§6 **三族插槽并存**，单次变更需在全量插槽中甄别适用项 |
| `tasks-template.md` | 212 | 四族整块装配 + Agent 角色表 + section 启用条件清单 |
| `adr.md` | 89 | 9 维技术栈评估表等重结构 |
| `spec-delta.md` | 77 | 强制 EARS + Gherkin 围栏块 |

即便一次轻量变更，也要在 758 行的 design 模板中穿行，生成与维护成本高。

**痛点 4：AI 在不同会话中经常遗忘项目情况、编程规范、踩过的坑。**

skill 内无记忆子系统。跨会话恢复完全依赖 AI「主动去读」持久文档，而哪些文档该读、读到什么程度没有机制保障；编程规范和踩坑经验没有沉淀载体，每次会话从零开始。`AGENTS.md` 承载仓库规则，但不承载项目级编码规范与经验教训。

### 1.3 为什么现在重构

调研显示 Trellis 已在四个痛点对应的方向上提供了成熟机制：workflow.md 单文件契约 + 面包屑每轮注入、inject-subagent-context 拦截式上下文注入、轻量 per-task 工件（prd.md / implement.md / jsonl manifest）、spec 自更新库 + workspace journal 跨会话记忆、trellis mem 跨会话对话检索。本地设计思路笔记也已给出了 prd/implement 模板与行为规约同步逻辑。时机成熟，可系统重构而非补丁式修改。

## 二、目的

### 2.1 重构目标

1. **自动化上下文注入**：subagent 启动前由运行时 hook 自动注入任务工件与 spec，编排器不再「靠自觉」。
2. **工作流可感知**：引入 workflow.md 单文件契约 + 面包屑机制 + `/specflow:continue` 命令，AI 每轮自动获知当前阶段与下一步，降低上手心智负担。
3. **模板瘦身**：取消 roadmap / 架构图 / ADR 阶段，design 精简为 Trellis per-task 格式，prd 与 implement 模板替换为本地笔记定义的轻量模板，单次变更上下文体量大幅下降。
4. **跨会话记忆闭环**：specflow 内建 spec（自更新规范库，仿 Trellis 通过 index.md 发现）+ journal（会话日志）+ requirements（行为规约派生产物）+ mem（跨会话对话检索），让 AI 跨会话可恢复项目情况、规范与踩坑。
5. **配置驱动 / 扩展点**：agent 类型与第三方 agent 接入通过配置声明，新增 agent 不改代码，避免硬编码行为。
6. **规避 Trellis 已知缺陷**：面包屑注入采用非破坏性方式，避免 issue #367 的 user message 污染问题。
7. **版本管理中立**：同时支持 git 和 jj，init 时自动扫描项目检测（优先 jj），检测不到再询问。

### 2.2 非目标

- 不追求一次性支持 14+ 平台，首期以 OpenCode 为主，预留多平台扩展点。
- 不考虑存量项目迁移（旧 specflow 产物视为只读历史，新工作流在 `.specflow/` 全新开始）。
- 不在本期实现 omo-slim 多 agent 集成的全部细节，仅实现扩展点与配置点；何时集成由用户在目标项目决定。
- 无 hook 则无自动化（接受降级为纯文档模式，不额外构建兜底）。

## 三、调研结论

### 3.1 Trellis 源码关键机制

**workflow.md 单文件契约 + 面包屑**：`.trellis/workflow.md` 用 `[workflow-state:STATUS]...[/workflow-state:STATUS]` 标签块定义每个阶段的面包屑文案。`inject-workflow-state.js` 插件每轮解析 workflow.md，按 `task.json.status` 匹配标签块，注入 `<workflow-state>...</workflow-state>`。关键设计：**hook 只解析，不存文案副本**，找不到匹配块时降级为固定一行 `Refer to workflow.md for current step.`<sup><a href="#cite-4">[4]</a></sup>。

**subagent 上下文注入**：`inject-subagent-context.js` 注册 `tool.execute.before` hook，拦截 `task` 工具调用。对 `implement`/`check`/`research` 三种 subagent，按固定顺序读取 jsonl manifest → `prd.md` → `design.md`（如存在）→ `implement.md`（如存在），拼装上下文后**原地修改 `args.prompt`** 注入。任务解析优先级：会话 runtime context → dispatch prompt 中的 `Active task:` hint → 单会话回退。

**任务状态机**：`task_store.py` 通过 `task.py create/start/finish/archive` 子命令流转 `task.json.status`（planning → in_progress → completed），状态变更同时触发面包屑切换与生命周期 hook。

**父子任务**：`task.py create --parent <父任务>` 建子任务；父任务 `children` 列表追加，子任务 `parent` 字段被设置；`task.py list` 缩进显示子任务并打 `[已完成/总数]`<sup><a href="#cite-7">[7]</a></sup>。

**跨会话记忆**：`.trellis/spec/`（按 package/layer 组织，每个目录有 `index.md` 入口 + Pre-Development Checklist）+ `.trellis/workspace/<developer>/journal-N.md`（每文件最多 2000 行，超限轮转）+ `.trellis/.runtime/sessions/<session-key>.json`（gitignored 的 session 活跃任务指针）。

**trellis mem**：读取本地 `~/.claude/projects/`、`~/.codex/sessions/`、`~/.pi/agent/sessions/` 的原始 JSONL 对话日志，按 `--phase brainstorm|implement|all` 检索，不上传任何内容。

### 3.2 Trellis 命令体系（everyday-use 文档）

Trellis 0.5+ 走 **skill-first**：大部分能力做成 auto-trigger skill，平台根据上下文自动触发。只保留会话边界 slash 命令<sup><a href="#cite-7">[7]</a></sup>：

| 类型 | 名称 | 触发 | 用途 |
|------|------|------|------|
| Command | `/trellis:start` | 手动（hook 平台自动注入） | 开启会话、加载上下文、任务分类 |
| Command | `/trellis:continue` | 手动 | 当前任务内推进下一步，AI 根据 status + 面包屑判断该走哪个 phase/step，用户不用记 workflow |
| Command | `/trellis:finish-work` | 手动，commit 落地后 | 归档 task + 写 session journal；工作区有未提交改动时拒绝执行 |
| CLI | `trellis upgrade` | 手动 | 升级全局 CLI package（②→①） |
| CLI | `trellis update` | 手动 | 把当前项目同步到本地 CLI 版本（③→②） |

三版本体系：①已发布版（npm 最新）②本地 CLI（全局安装）③项目版（`.trellis/` 模板版本）。完整升级 = 先 `upgrade`（CLI）再 `update`（项目）<sup><a href="#cite-7">[7]</a></sup>。

`/trellis:continue` 是单任务内推进：AI 读 task status + 面包屑对照 workflow.md 判断当前 phase/step，推进到下一步。典型流程：描述需求→brainstorm 建 task 起 prd→`continue`→判断是否补 design/implement→`continue`→start 进 implement/check→`continue`→update-spec→finish-work<sup><a href="#cite-7">[7]</a></sup>。

5 个 auto-trigger skill：`trellis-brainstorm`（澄清需求起 PRD）、`trellis-before-dev`（编码前读 spec）、`trellis-check`（验证+自修复）、`trellis-update-spec`（沉淀经验到 spec）、`trellis-break-loop`（bug 根因 5 维分析）<sup><a href="#cite-7">[7]</a></sup>。

### 3.3 Trellis issue #367（面包屑注入缺陷）

issue #367 报告：Trellis 在 Pi 平台用 `input` 事件回调的 `transform` 动作把压缩的 trellis 上下文**注入到 user message 本身**，导致用户消息对话框被大段 trellis 上下文占据，破坏可读性<sup><a href="#cite-8">[8]</a></sup>。维护者确认后续应「从 user message 换成 custom message」解决。该缺陷在 OpenCode 中经用户使用确认依然存在。

**对 specflow 的教训**：面包屑与 session-start 注入必须采用**非破坏性方式**——注入内容作为独立 context/system 消息或 custom message，绝不能修改 user message 本身。详见 5.2.1 节的规避设计。

### 3.4 本地设计思路笔记

笔记定义了与 Trellis 兼容的轻量工件体系：

- **workflow.md 只管流程调度**，不内嵌模板，只引用 skill。
- **prd.md 是唯一事实来源**，`spec/requirements/` 是派生输出（只读、按领域组织、增删改全部通过 Phase 3 同步）。
- **需求 ID 全局统一** `REQ-<DOMAIN>-NNN`。
- prd 模板分单任务（EARS + Gherkin / 度量约束）与父任务（子任务映射 + 跨子任务验收）。
- implement 模板分 TDD 行为切片（RED/GREEN/REFACTOR）与度量驱动。
- 多 agent 集成走独立插件，不修改核心代码。

### 3.5 可借鉴 vs 不照搬

| Trellis 机制 | 借鉴 | 调整理由 |
|--------------|------|----------|
| workflow.md + 面包屑 | 直接借鉴，并规避 #367 | 解决痛点 2；注入改非破坏性 |
| inject-subagent-context 拦截注入 | 借鉴架构，jsonl 动态加载 | 解决痛点 1；上下文顺序由 jsonl 动态决定，不写入配置 |
| per-task prd.md / implement.md / jsonl | 直接借鉴 + 笔记模板 | 解决痛点 3 |
| spec 自更新库（index.md 发现）+ journal + mem | 借鉴，内建到 specflow | 解决痛点 4 |
| Python task.py CLI | 改用 Go | 单二进制分发，无运行时依赖 |
| continue / finish-work / upgrade / update | 直接借鉴 | 版本管理与用户体验 |
| auto-trigger skill 机制 | 直接借鉴 | 降低学习成本 |
| 14 平台适配层 | 不照搬 | 首期 OpenCode 为主，预留扩展点 |
| channel runtime / multi-agent forum | 不照搬 | 本期不做 |

## 四、总体思路

### 4.1 设计哲学

1. **契约与实现分离**：workflow.md 是流程唯一事实源，插件只解析不存逻辑；agent 行为由配置声明，代码只提供扩展点。
2. **状态驱动而非文件驱动**：阶段由 `task.json.status` 驱动面包屑切换，不再靠文件存在性猜测阶段。
3. **上下文窄而准**：上下文文件清单与读取顺序全部由 jsonl manifest 声明，脚本按 jsonl 动态加载，不设固定契约、不写入配置文件。
4. **记忆分层持久**：任务事实 → `changes/`，团队规范 → `spec/`，行为规约 → `spec/requirements/`，会话日志 → `workspace/journal`，对话历史 → `mem`，session 运行时 → `.runtime/`（gitignored）。
5. **中文优先**：specflow 产出的所有 md 文档（含 `.specflow/` 下）除技术术语外使用中文表述。
6. **版本管理中立**：git 与 jj 并存，配置选择。

### 4.2 架构总览

```
目标项目
├── .opencode/
│   ├── skills/                        ← skill 层（specflow 内置 skills，init 同步到此）
│   │   ├── specflow-brainstorm/SKILL.md
│   │   ├── specflow-before-dev/SKILL.md
│   │   ├── specflow-check/SKILL.md
│   │   ├── specflow-update-spec/SKILL.md
│   │   ├── specflow-break-loop/SKILL.md
│   │   ├── specflow-sync-requirements/SKILL.md
│   │   ├── specflow-continue/         ← /specflow:continue command
│   │   └── specflow-finish-work/      ← /specflow:finish-work command
│   ├── plugins/                       ← OpenCode 插件层（hook 自动化，JS）
│   │   ├── session-start.js
│   │   ├── inject-workflow-state.js
│   │   └── inject-subagent-context.js
│   └── agents/                        ← 平台 agent 定义（native + custom 同步到此；platform 不同步）
│       ├── specflow-implement.md      ← native：specflow 内置，init 同步
│       ├── specflow-oracle.md         ← custom：用户自建，sync-agent 同步
│       └── ...                        ← platform 类不出现（平台自带）
└── .specflow/                          ← 运行时状态层（目标项目内，md 中文表述）
    ├── .developer                      ← 开发者身份（gitignored，init -u 写入）
    ├── .vcs                            ← 版本管理选择：git | jj（gitignored，init 询问）
    ├── workflow.md                     ← 工作流契约（面包屑 + 路由 + Phase 定义）
    ├── config.yaml                     ← 项目级配置
    ├── agents.yaml                     ← agent 声明（source / jsonl / 约束）
    ├── agents/                         ← 自建 agent 定义（source: custom）
    │   └── <name>.md
    ├── spec/                           ← 自更新规范库（仿 Trellis，index.md 发现）
    │   ├── index.md                    ← 顶层索引（Pre-Development Checklist）
    │   ├── <layer>/index.md            ← 分层索引（backend / frontend / guides…）
    │   ├── <layer>/*.md                ← 编码约定、guides
    │   └── requirements/               ← 行为规约（prd.md 派生，按领域）
    │       ├── index.md
    │       └── <domain>.md
    ├── changes/                        ← 变更任务工件（保留 specflow changes/ 语义）
    │   ├── <change-id>/                ← change-id = YYYY-MM-DD-short-slug-N
    │   │   ├── task.json               ← 元数据 + status
    │   │   ├── prd.md                  ← 需求（含意图段）+ EARS/Gherkin
    │   │   ├── implement.md            ← 执行计划（TDD/度量）
    │   │   ├── design.md               ← 可选，Trellis per-task 格式
    │   │   ├── implement.jsonl         ← implement agent 上下文清单（含顺序）
    │   │   ├── check.jsonl             ← check agent 上下文清单（含顺序）
    │   │   └── research/               ← 研究产物（可选）
    │   └── archive/<YYYY-MM>/          ← 归档任务
    ├── workspace/<developer>/          ← 跨会话 journal
    │   ├── journal-N.md
    │   └── index.md
    ├── .fingerprints.json              ← 管理文件 hash（gitignored，update 冲突检测用）
    ├── .update-candidates/             ← update 冲突时的新版本候选文件（gitignored）
    └── .runtime/sessions/              ← session 指针（gitignored）
```

三层职责：

| 层 | 载体 | 职责 | 是否进 git |
|----|------|------|-----------|
| skill 层 | `.opencode/skills/`（OpenCode） | specflow 内置 auto-trigger skills + commands（know-how、模板、审查规则） | 是 |
| 插件层 | `.opencode/plugins/` | 自动化注入（面包屑 / subagent 上下文 / session-start） | 是 |
| agent 层 | `.opencode/agents/` | native（specflow 内置）+ custom（自建）agent 定义；platform 类不同步 | 是 |
| 状态层 | `.specflow/` | 任务状态、工件、spec、journal、运行时 | spec/changes/workspace 进 git；.runtime/.developer 不进 |

### 4.3 已确认的核心决策

| 决策 | 选择 | 来源 |
|------|------|------|
| 运行时层 | 引入脚本 + hook 层 | 用户确认 |
| 目标平台 | OpenCode 为主（JS 插件 hook） | 用户确认 |
| 模板瘦身 | 取消 roadmap/架构图/ADR；design 精简为 Trellis per-task；prd/implement 用笔记模板 | 用户确认 |
| 记忆机制 | specflow 内建 spec（index.md 发现）+ journal + mem | 用户确认 |
| 实现风格 | 配置 / 扩展点，避免硬编码 | 用户确认 |
| 脚本语言 | Go（CLI 层）；平台特定领域用平台要求语言（OpenCode 插件用 JS） | 用户确认 + 5.1 评估 |
| 任务目录 | 保留 specflow `changes/` 语义，change-id = `YYYY-MM-DD-short-slug-N` | 用户确认 |
| 上下文加载 | jsonl manifest 声明文件与顺序，脚本动态加载，不写入配置、不设固定契约 | 用户确认 |
| 第三方 agent | source: native（specflow 内置）/ source: platform（复用平台已有）/ source: custom（自建） | 用户确认 |
| skill 放置 | specflow 内置 skill 随平台指定放到平台目录（OpenCode → `.opencode/skills/`） | 用户确认 |
| 行为规约同步 | 新增 skill 实现，workflow 只负责 skill 引入与调度 | 用户确认 |
| 意图信息 | 保留简陋意图段合并到 prd，不参与行为规约同步 | 用户确认 |

### 4.4 关于 specflow-workflow/SKILL.md 是否保留（疑问答复）

**结论：不再保留单一的 `specflow-workflow/SKILL.md`。**

重构后采用 Trellis 的 skill-first 架构，能力拆分为多个独立 auto-trigger skill，每个 skill 有自己的 `SKILL.md`：

- 工作流契约由 `.specflow/workflow.md` 承担（面包屑 + 路由 + Phase 定义）。

- 模板由各 skill 的 `assets/` 承担。
- 旧的单体 `SKILL.md` 的路由/审查/约束职责分散到 workflow.md 与各 skill 中。

## 五、可能的方案（详细设计）

### 5.1 运行时层：Go CLI + OpenCode 插件

#### 5.1.1 脚本语言评估（Go vs Python vs JS）

| 维度 | Go | Python | JS (Node) |
|------|-----|--------|-----------|
| 分发方式 | **单二进制**，无运行时依赖，跨平台编译 | 需 python 运行时 + pip | 需 node 运行时 + npm |
| `specflow init` 全局可用 | `go install` 或下载二进制即用 | pipx/pip install，依赖管理复杂 | npm install -g |
| 被 OpenCode 插件调用 | exec 调用二进制，干净解耦 | exec 调用 python 脚本 | exec 或直接 require |
| JSON/YAML 处理 | 标准库 json + yaml.v3，够用 | 最灵活 | json 原生 + js-yaml |
| 开发迭代 | 需编译（go build 快），类型安全 | 快速迭代，类型弱 | 快速迭代，与插件同语言 |

**结论：Go 倾向合理，采纳。** 核心理由：specflow 需要独立全局 CLI（`specflow init`），单二进制分发无需预装运行时；OpenCode 插件通过 `exec` 调用 Go 二进制完全解耦（OpenCode 工具层已验证通过 `exec(params.command)` 执行外部命令<sup><a href="#cite-9">[9]</a></sup>）；Go 类型安全适合稳定分发的 CLI。平台特定领域（OpenCode 插件）仍用 JS。

#### 5.1.2 插件层（OpenCode JS 插件）

三个插件各 hook 一个事件，**逻辑薄，只做 hook 拦截 + exec 调用 + 结果注入**：

| 插件 | hook 事件 | 触发频率 | 职责 |
|------|-----------|----------|------|
| `session-start.js` | `chat.message`（首消息去重） | 每会话一次 | exec 调用 `specflow get-context`，注入 session context |
| `inject-workflow-state.js` | `chat.message` | 每轮 | 解析 workflow.md，按 task status 注入面包屑（**非破坏性**，见 5.2.1） |
| `inject-subagent-context.js` | `tool.execute.before` | 每次 task 工具调用 | 拦截 subagent 派发，exec 调用 `specflow build-context` 取上下文注入 prompt |

#### 5.1.3 Go CLI 子命令

| 命令 | 作用 |
|------|------|
| `specflow init -u <name> [--opencode\|--pi] [--vcs git\|jj]` | 安装三层到目标项目（见 5.8），VCS 自动扫描检测 |

| `specflow task start` | status → in_progress；**检查 session 独占**：若其他 session 已指向同一任务则拒绝并提示占用方信息 |

| `specflow task archive` | status → completed，移目录到 archive/，auto-commit |

| `specflow task add-subtask <parent> <child>` | 链接父子任务 |
| `specflow task remove-subtask <parent> <child>` | 解除父子关联 |

| `specflow add-session --title --summary` | 追加 journal 条目 |
| `specflow validate` | 校验 config/agents/jsonl 引用完整性 |
| `specflow doctor` | 检查配置健康，引导修复（见 5.9） |

| `specflow update` | 项目同步到本地 CLI 版本（③→②，含文件指纹冲突检测，见 5.10） |

### 5.2 workflow.md 单文件契约 + 面包屑

#### 5.2.1 面包屑注入与 #367 规避

issue #367 的根因是**把注入内容写进 user message 本身**，导致 UI 污染<sup><a href="#cite-8">[8]</a></sup>。specflow 的规避设计：

- **inject-workflow-state.js 采用非破坏性注入**：注入内容作为独立的结构化块附加到上下文，不修改 user message 原文。优先用 OpenCode 提供的 context/system message 注入通道；若仅能修改 prompt，则把面包屑块放在 user message **之前**的独立段落并用明确分隔标记，且不压缩、不替换用户原文。
- **session-start.js 同理**：session context 作为独立块注入，不污染 user message。
- **设计约束**：任何注入都不得 `transform` 用户原文，只允许前置附加或独立通道。

> 待决：OpenCode `chat.message` hook 的确切注入 API（能否插入独立 message、还是只能修改 prompt）在实现时确认，以选择最干净的非破坏性通道。

#### 5.2.2 workflow.md 结构

安装在目标项目 `.specflow/workflow.md`（中文表述），可 fork：

```markdown
## 阶段索引
- Phase 1: 规划  (status=planning)
- Phase 2: 执行  (status=in_progress)
- Phase 3: 收尾  (status=completed)

[workflow-state:no_task]
无活跃任务。先分类请求：简单对话 / inline 小任务 / 完整 specflow 任务。
建任务需取得用户确认（task creation consent gate）。
[/workflow-state:no_task]

[workflow-state:planning]
流程：brainstorm → 编写 prd.md → 编写 implement.md → (可选) design.md → curate jsonl → task start
加载 `specflow-brainstorm` 进行需求探索。
[/workflow-state:planning]

[workflow-state:in_progress]
流程：implement sub-agent → check sub-agent → 加载 `specflow-sync-requirements` → spec update → commit → finish-work
Dispatch prompt 以 `Active task: <path>` 开头。
上下文：按 jsonl manifest 条目顺序加载。
[/workflow-state:in_progress]

### 活跃任务路由
- 规划或需求不清 -> `specflow-brainstorm`。
- in_progress 实现/验收 -> 派发 implement / check sub-agent。
- 行为规约同步 -> `specflow-sync-requirements`。
- 重复调试 -> `specflow-break-loop`；spec 更新 -> `specflow-update-spec`。
```

**面包屑工作原理**（同 Trellis，注入方式按 5.2.1 规避）：`inject-workflow-state.js` 用正则解析 `[workflow-state:STATUS]` 标签块，按 `task.json.status` 匹配。**脚本内不存 fallback 字典**，找不到匹配块时降级为 `Refer to workflow.md for current step.`。改面包屑文案 = 改 markdown，下一条消息即生效<sup><a href="#cite-4">[4]</a></sup>。

**workflow 只负责 skill 引入与调度**：面包屑和路由表只声明「加载哪个 skill」，具体逻辑（如行为规约同步的增删改规则）在 skill 内实现。

### 5.3 subagent 上下文自动注入（jsonl 动态加载 + 第三方 agent）

#### 5.3.1 上下文加载机制（jsonl 声明文件与顺序，无固定契约）

上下文文件清单与读取顺序**全部由 jsonl manifest 声明**，脚本按 jsonl 动态加载：

- jsonl manifest 是文件引用清单，每行 `{"file":"<path>","reason":"<说明>"}`。需要加载的 prd.md、design.md、implement.md 以及 spec/research 文件**全部写入 jsonl**，按 jsonl 内的行顺序逐条加载。
- **不设固定契约顺序**（不硬编码「先 jsonl 再 prd 再 design 再 implement」），顺序由 jsonl 条目排列决定。
- jsonl 放 spec 文件、research 文件、任务工件（prd/design/implement），**不放马上要修改的源文件**。
- `specflow task create` seed 出带 `{"_example":"..."}` 自描述行的 jsonl，AI 在 Phase 1.5 curate 真实条目（含工件引用）。
- `specflow build-context <agent> --task <dir>` 由 Go CLI 实现：读对应 jsonl → 按条目顺序拼装上下文 → 输出给插件注入 prompt。

#### 5.3.2 第三方 agent 接入（agents.yaml + 扩展点）

agents.yaml 声明 agent 元信息，支持三种 source：

```yaml
agents:
  # native：specflow 内置 agent（定义随 CLI 打包，init/update 时同步到平台 agents 目录）
  specflow-implement:
    source: native
    jsonl_file: implement.jsonl
    require_task: true
    constraints:
      - "禁止 git commit / push / merge"

  specflow-check:
    source: native
    jsonl_file: check.jsonl
    require_task: true
    can_write: true
    constraints:
      - "可修复 finding 但禁止 git 操作"

  # platform：复用第三方平台已有 agent（平台自带定义，specflow 只注入上下文，不同步）
  code-reviewer:
    source: platform
    jsonl_file: code-reviewer.jsonl
    require_task: false
    constraints:
      - "只读审查"

  # custom：自建 agent（定义在 .specflow/agents/<name>.md，需 sync-agent 同步到平台目录）
  specflow-oracle:
    source: custom
    jsonl_file: oracle.jsonl
    agent_file: agents/oracle.md
    require_task: true
    constraints:
      - "只读分析，禁止改源码"

  specflow-research:
    source: native
    jsonl_file: null
    readonly: true
```

**三种 source 的区别**：

| source | agent 定义来源 | jsonl | 同步 | 适用场景 |
|--------|---------------|-------|------|---------|
| `native` | specflow 内置（随 CLI 打包，`go:embed`） | 指定 jsonl 文件名 | `specflow init` / `specflow update` 自动同步到平台 agents 目录 | specflow 提供的内置 agent（如 specflow-implement） |
| `platform` | 平台自带（specflow 不管定义） | 指定 jsonl 文件名 | **不同步**（平台已有定义） | 复用第三方平台已有 agent，specflow 只注入上下文 |
| `custom` | `.specflow/agents/<name>.md` | 指定 jsonl 文件名 | `specflow sync-agent` 同步到平台目录 | 用户自建 agent，定义与上下文都由 specflow 管 |

**插件逻辑（伪代码）**——直接用 `subagent_type` 原名查配置，不做前缀替换：

```js
// inject-subagent-context.js
export default async ({ directory }) => {
  const config = loadAgentsConfig(directory)         // 读 agents.yaml
  return {
    "tool.execute.before": async (input, output) => {
      if (input.tool !== "task") return
      const subagentType = output.args.subagent_type  // 原名，如 specflow-implement
      const agentConf = config.agents[subagentType]   // 直接查配置
      if (!agentConf) return                          // 配置未声明则不处理（扩展点）
      // exec 调用 Go CLI 构建上下文（按 jsonl 动态加载，CLI 内完成）
      const context = await exec(["specflow", "build-context", subagentType,
                                  "--task", resolveTaskDir(...)])
      output.args.prompt = wrapPrompt(subagentType, output.args.prompt,
                                       context, agentConf.constraints)
    }
  }
}
```

**扩展点（omo-slim 等第三方集成）**：新增 agent = 在 agents.yaml 加一条配置（source: native / platform / custom）+ 可选在 `.specflow/agents/` 加定义文件（custom 类）+ 可选 seed 对应 jsonl。不改插件代码、不改 Go CLI。

**任务解析**沿用 Trellis 三级优先级：会话 runtime context（`input.sessionID`）→ dispatch prompt 中 `Active task: <path>` hint → 单会话回退。

#### 5.3.3 agent 同步到平台

**native agent 同步**：specflow 内置 agent 定义随 Go CLI 打包（`go:embed`）。`specflow init` 和 `specflow update` 时自动将所有 `source: native` 的 agent 定义同步到平台 agents 目录。

**custom agent 同步**：`specflow sync-agent <name>`（Go CLI）将 `.specflow/agents/<name>.md` 同步到平台 agent 目录。`specflow init` 和 `specflow update` 时也会自动同步所有 custom agent。

**platform agent 不同步**：`source: platform` 的 agent 定义由平台自带，specflow 不负责同步，只通过 jsonl 注入上下文。

| source | 同步时机 | 同步来源 | 同步目标（OpenCode） |
|--------|---------|---------|---------------------|
| `native` | init / update | Go CLI 内嵌（`go:embed`） | `.opencode/agents/<name>.md` |
| `platform` | 不同步 | — | — |
| `custom` | init / update / `sync-agent` | `.specflow/agents/<name>.md` | `.opencode/agents/<name>.md` |

| `spec/requirements/index.md` | 行为规约索引 | specflow-sync-requirements 维护 |
| `spec/requirements/<domain>.md` | 行为规约（prd.md 派生，按领域） | Phase 3.1 specflow-sync-requirements |

**spec 发现机制（index.md）**：仿 Trellis，每个 spec 目录有 `index.md` 入口，含 Pre-Development Checklist（指向具体指南文件）。`specflow get-context` 与 `specflow build-context` 扫描 spec 目录树，通过 `index.md` 发现可用规范；jsonl manifest 引用具体 spec 文件路径。`specflow-before-dev` skill 在编码前读受影响领域的 spec 索引。

**自更新闭环**：Phase 3.2 加载 `specflow-update-spec`，判断是否产出可复用规则，写回 `spec/<layer>/`，必要时更新 index.md。

#### 5.5.2 journal 会话日志

`.specflow/workspace/<developer>/journal-N.md`（进 git）：每次 finish-work 用 `specflow add-session` 追加 session 总结；每文件最多 2000 行（`config.yaml:max_journal_lines`），超限轮转。

#### 5.5.3 developer 身份与 session 运行时

- `.specflow/.developer`（gitignored）：`specflow init -u <name>` 写入，用于 journal 隔离与 session 指针。
- `.specflow/.runtime/sessions/<session-key>.json`（gitignored）：把一个 AI session 指向一个任务，同一仓库不同窗口可做不同任务。

#### 5.5.4 mem 跨会话对话检索（仿 trellis mem）

`specflow mem` 读取本地 AI 平台的原始对话日志（OpenCode 会话 JSONL 等），支持跨会话检索历史对话，不上传任何内容：

| 子命令 | 作用 |
|--------|------|
| `specflow mem list` | 列出可检索的项目与会话 |
| `specflow mem search <query>` | 按关键词检索历史对话 |
| `specflow mem context <query>` | 检索并输出上下文片段 |
| `specflow mem extract` | 提取指定会话内容 |

按 `--phase brainstorm|implement|all` 在任务边界切片检索（仿 Trellis）。让 AI 能回答「上次怎么解的」「之前讨论过吗」。

**AI 跨会话恢复路径**：session-start 注入（开发者 + git/jj + 活跃任务 + Next-Action + spec 索引 + journal）→ 每轮面包屑提醒阶段 → 任务工件持久化在 changes/ → 项目约定在 spec/（index.md 发现）→ 历史决策用 journal 回忆 → 细节对话用 mem 检索。

### 5.6 配置与扩展点（避免硬编码）

| 硬编码风险点 | 配置化 / 扩展点方案 |
|-------------|-----------|
| 插件写死 agent 类型 | `agents.yaml` 声明 agents 键，插件遍历配置 |
| 上下文读取顺序 | jsonl manifest 声明文件与顺序，脚本动态加载 |
| 面包屑文案 | workflow.md 标签块为唯一来源，脚本只解析 |
| agent 路由 | workflow.md `活跃任务路由` 表 |
| 约束文案 | `agents.yaml` 的 `constraints` 列表注入 prompt |
| 第三方 agent 接入 | `source: native` / `source: platform` / `source: custom` + `.specflow/agents/<name>.md`（custom 类） |
| 版本管理 | `config.yaml:vcs` 选择 git/jj |

**config.yaml 关键项**：`vcs`（git|jj）、`max_journal_lines`、`session_auto_commit`、`hooks.after_create/after_start/after_finish/after_archive`、`packages`/`default_package`（monorepo）。

### 5.8 安装：specflow init 命令

```bash
specflow init -u <developer> [--opencode|--pi] [--vcs git|jj]
```

**init 做什么**（Go CLI 实现，安装三层）：

1. **VCS 自动检测**：扫描项目根目录——存在 `.git/` 则检测为 git，存在 `.jj/` 则检测为 jj。若两者都存在，优先 jj（jj 可同时管理 git 仓库）。若都不存在，交互询问用户选择 git 或 jj。`--vcs` 参数可覆盖检测结果。写入 `config.yaml:vcs` 和 `.specflow/.vcs`。
2. 在目标项目创建 `.specflow/`，seed `workflow.md` / `config.yaml` / `agents.yaml` / `spec/` seed（index.md，无引导任务）。
3. 写入 `.specflow/.developer`（开发者身份，gitignored）。
4. 读取 `platforms/<platform>/install-map.yaml`，按映射复制 specflow 内置 skill 层到平台 skill 目录（`--opencode` 时 → `.opencode/skills/`）。
5. 复制平台插件到对应目录（`--opencode` 时 → `.opencode/plugins/`）。
6. 同步 native agent 定义到平台 agents 目录（从 Go CLI 内嵌资源）。
7. 同步 custom agent 到平台 agents 目录（如有）。
8. **记录文件指纹**：对所有 specflow 管理的文件计算 hash，写入 `.specflow/.fingerprints.json`（gitignored）。
9. 提示用户重启 AI Agent。

### 5.9 配置校验与 doctor

**配置校验失败处理**：阻断任务并告警原因。`specflow validate` 校验 config.yaml / agents.yaml / jsonl 引用完整性（引用文件是否存在、格式是否正确）。

**`specflow doctor` 命令**：检查配置健康，尝试诊断并引导修复：
- 检查 `.specflow/` 结构完整性（workflow.md / config.yaml / agents.yaml 是否存在且有效）。
- 检查 agents.yaml 中 source: custom 的 agent_file 是否存在。
- 检查 jsonl 引用文件是否存在。
- 检查 native agent 是否已同步到平台目录（init/update 是否执行过）。
- 检查 custom agent 是否已同步到平台目录。
- 检查 `.fingerprints.json` 是否存在且与当前管理文件一致（检测手动修改）。
- 检查 `.update-candidates/` 目录是否有待处理的新版本文件（提醒用户手动合并）。
- 检查 stale session 指针（长期未活动的 session 占用任务）。
- 输出诊断报告与修复建议。

### 5.10 版本管理与升级（仿 Trellis 三版本体系）

三版本：①已发布版（GitHub Releases / brew）②本地 CLI（全局 specflow 二进制）③项目版（`.specflow/` 模板版本）。

| 命令 | 作用 |
|------|------|
| `specflow upgrade` | 升级全局 CLI 二进制（②→①），跟随 channel（latest/beta） |
| `specflow update` | 把当前项目 specflow 管理文件同步到本地 CLI 版本（③→②），含 workflow.md / 插件 / skill / native agent 更新 |

完整升级 = 先 `upgrade`（CLI）再 `update`（项目）。

#### 5.10.1 specflow update 更新内容

`specflow update` 将以下 specflow 管理文件同步到本地 CLI 内嵌的最新版本：

| 类别 | 管理文件（update 覆盖） | 用户文件（update 不动） |
|------|------------------------|----------------------|
| 模板 | `.specflow/workflow.md` | `.specflow/config.yaml`（仅追加新键）、`.specflow/agents.yaml`（仅更新 native 条目，保留 custom 条目） |
| skill | `.opencode/skills/specflow-*/**` | — |
| 插件 | `.opencode/plugins/*.js` | — |
| native agent | `.opencode/agents/specflow-*.md` | `.opencode/agents/<custom>.md`（用户自建） |
| spec seed | `.specflow/spec/index.md`（仅首次 seed，后续不覆盖） | `.specflow/spec/**` 其余内容（用户编码约定） |

#### 5.10.2 文件指纹机制（防止覆盖用户修改）

为防止用户本地修改过的文件被 `specflow update` 静默覆盖，引入文件指纹（hash）机制：

**指纹存储**：`.specflow/.fingerprints.json`（gitignored），结构：

```json
{
  "specflow_version": "0.1.0",
  "files": {
    ".specflow/workflow.md": "sha256:abc123...",
    ".opencode/plugins/inject-workflow-state.js": "sha256:def456...",
    ".opencode/skills/specflow-brainstorm/SKILL.md": "sha256:ghi789...",
    ".opencode/agents/specflow-implement.md": "sha256:jkl012..."
  }
}
```

**update 流程**（每个管理文件逐个处理）：

```
1. 读取 .fingerprints.json 中该文件上次记录的 hash（install/update 时写入）
2. 计算该文件当前内容的 hash
3. 比较三者：旧指纹 vs 当前文件 vs CLI 新版本
   ├─ 当前 == 旧指纹 → 用户未改 → 直接覆盖为新版本 ✅
   ├─ 当前 != 旧指纹 且 新版本 == 旧指纹 → 用户改了但 CLI 没更新 → 保留用户版本，跳过 ✅
   └─ 当前 != 旧指纹 且 新版本 != 旧指纹 → 用户改了且 CLI 也更新了 → 冲突 ⚠️
```

**冲突处理**（上述第三种情况）：交互询问用户：

| 选项 | 行为 |
|------|------|
| 覆盖 | 用 CLI 新版本覆盖用户修改（用户修改丢失） |
| 合并 | 保留用户版本，将 CLI 新版本写入 `.specflow/.update-candidates/<path>` 供用户手动合并 |
| 放弃 | 跳过该文件，保留用户版本，不写入新版本 |

**merge 策略说明**：specflow 不做自动合并（md/js 自动合并风险高）。选择「合并」时只是把新版本放到候选目录，提示用户手动 diff。`specflow doctor` 会检测 `.update-candidates/` 目录并提醒用户处理。

**update 完成后**：刷新 `.fingerprints.json`，更新所有已覆盖文件的 hash 为新版本内容。

**分发渠道**：GitHub Releases 二进制（macOS arm64/amd64、Linux、Windows）+ Homebrew formula（`brew install specflow`）+ `go install`。

### 5.11 模板分享机制

specflow 支持模板分享，模板内容包括：

| 模板类型 | 内容 | 分享方式 |
|----------|------|---------|
| spec 模板 | spec/ 下的编码约定、guides 结构 | registry 拉取 |
| workflow 模板 | workflow.md 面包屑与路由 | registry 拉取 |

### 5.11.1 命令与 auto-trigger skills 汇总

仿 Trellis skill-first 架构<sup><a href="#cite-7">[7]</a></sup>：

| 类型 | 名称 | 触发 | 用途 |
|------|------|------|------|
| Command | `/specflow:continue` | 手动 | 当前任务内推进下一步，AI 根据 status + 面包屑判断该走哪个 phase/step |
| Command | `/specflow:finish-work` | 手动，commit 落地后 | 归档 task + 写 journal；工作区有未提交改动时拒绝执行 |

### 5.12 版本管理中立（git + jj）

**VCS 自动检测**（init 时执行）：

1. 扫描项目根目录，检测 `.git/` 或 `.jj/` 目录是否存在。
2. 存在 `.jj/` → 检测为 jj（jj 可同时管理 git 仓库，优先级高）。
3. 仅存在 `.git/` → 检测为 git。
4. 两者都不存在 → 交互询问用户选择 git 或 jj（项目尚未初始化版本管理）。
5. `--vcs git|jj` 参数可覆盖检测结果。

检测结果写入 `config.yaml:vcs` 和 `.specflow/.vcs`（gitignored）。所有涉及版本管理的 CLI 操作根据 vcs 配置分派：

| 操作 | git 模式 | jj 模式 |
|------|---------|---------|
| 状态检查 | `git status --porcelain` | `jj status` |
| 提交 | `git add` + `git commit` | `jj describe` + `jj new` |
| 归档 auto-commit | `git commit -m "chore(task): archive ..."` | `jj describe -m "..."` + `jj new` |
| journal auto-commit | `git commit -m "chore: record journal"` | `jj describe -m "..."` + `jj new` |

遵循 `AGENTS.md` 的 jj 约定：本地工作流用 `jj describe` + `jj new`，不用 `git add/commit/stash/checkout`。

## 六、理由

| 决策 | 理由 |
|------|------|
| 引入脚本 + hook 层 | 痛点 1/2 的自动化无法靠纯文档实现；Trellis 已验证 hook 拦截式注入可行且解耦 |
| OpenCode 为主 | 用户主战场；OpenCode JS 插件 hook 机制成熟 |
| Go 作 CLI 语言 | 单二进制分发，`specflow init` 无需预装运行时；插件层 exec 调用解耦；类型安全（见 5.1.1） |
| workflow.md + 面包屑（非破坏性） | 每轮自动注入阶段状态；改流程=改 markdown；规避 #367 用独立通道不污染 user message<sup><a href="#cite-8">[8]</a></sup> |
| jsonl 动态加载上下文 | 仿 Trellis 成熟机制；上下文窄而准；文件与顺序由 jsonl 声明，无固定契约、不写入配置 |
| native/platform/custom 三 source | 用户要求；native = specflow 内置（随 CLI 打包同步），platform = 复用平台已有（不同步），custom = 自建并同步；omo-slim 等仅加配置不改代码 |
| 取消 roadmap/架构图/ADR | 用户确认；轻量变更中负担最重；长期技术决策由 spec 规范库 + design.md 承载 |
| design 用 Trellis per-task 格式 | 758 行三族并存是最大模板负担；per-task 可选 design.md 足够 |
| prd/implement 用笔记模板 | 笔记模板已设计且与 Trellis 兼容；总量降至 ~185 行 |
| spec index.md 发现 | 仿 Trellis 成熟机制；规范库可扩展、可发现；自更新闭环 |
| 内建 spec + journal + mem | 用户确认；spec 自更新 + journal 跨会话 + mem 对话检索，三层记忆闭环 |
| 行为规约同步用 skill | 用户确认；workflow 只管调度，逻辑在 skill |
| 保留 changes/ + change-id | 用户确认；保留 specflow 原有语义 |
| continue/finish-work/upgrade/update | 仿 Trellis；降低学习成本；版本管理可追踪 |
| auto-trigger skills | 仿 Trellis skill-first；平台自动匹配意图，用户零学习成本 |
| git + jj 中立 | 用户要求；自动扫描项目检测（优先 jj），检测不到再询问；遵循 jj 约定 |
| 不保留 SKILL.md 单体入口 | skill-first 架构下能力拆分为多 skill，workflow.md 承担契约 |
| doctor 命令 | 配置校验失败阻断 + 引导修复，提升可维护性 |
| 模板分享 | 用户要求；spec/workflow/agent 可复用 |
| brew formula | 用户确认；macOS 主流分发渠道 |
| 源仓库 platforms/ 平台区分 | 用户要求；插件 hook API 平台特定需隔离；新增平台只加目录不改 skills/agents 内容 |
| update 文件指纹 | 用户要求；防止覆盖用户修改；三路比对（旧指纹/当前/新版本）精确检测冲突 |

## 八、已确认的两个决策

### 8.1 task creation consent gate（建任务确认门禁）——已确认保留

**机制**：当 AI 判断用户的请求值得创建一个完整 specflow 任务时，**不直接创建，而是先询问用户「是否创建任务？」**，得到确认后才 `task create`。目的是防止 AI 对简单问题也兴师动众建任务（产生目录、task.json、prd.md 等开销）。

任务分类逻辑<sup><a href="#cite-7">[7]</a></sup>：

| 请求类型 | 判断依据 | 处理 |
|----------|---------|------|
| 简单对话 | 问答、解释、查询，不改仓库 | 默认不建任务 |
| Inline 小任务 | 当前回合可理解和验证的局部改动 | 只问本回合是否需要建任务，用户说不需要则 inline |
| 完整任务 | 多文件改动或需持久规划 | 询问是否创建 specflow task 并进入 planning |

**确认结果**：用户同意保留此机制。AI 在创建任务前必须先征得用户同意，简单对话/小任务可以 inline 不建任务。

### 8.2 同一任务多窗口并发冲突策略——已确认采用方案 B（session 独占指针）

**场景**：两个 AI 会话/窗口（如两个 OpenCode 窗口）同时指向同一个任务并尝试操作（写 prd.md、改 task.json、archive 等），可能产生文件冲突。

**确认结果**：用户选择**仅方案 B（session 独占指针）**，不采用推荐的 A+B 结合。

**机制设计**：

1. **独占规则**：同一任务同时只能被一个 session 指向。session 指针存储在 `.specflow/.runtime/sessions/<session-key>.json`（gitignored），记录该 session 当前活跃的任务路径。
2. **task start 检查**：`specflow task start` 执行时，检查目标任务是否已被其他 session 指向。若是，拒绝执行并提示「任务 X 正被 session Y 占用，请先 finish 或选择其他任务」。
3. **正常释放**：`specflow task finish` 清除 session 指针，释放任务独占，其他 session 可随后 start。
4. **Stale 指针处理**：session 异常退出（如终端关闭、进程崩溃）时指针不会自动释放。提供 `specflow task release <task-id>` 命令强制清除 stale 指针。`specflow doctor` 也会检测长期未活动的 stale 指针并提示清理。
5. **不使用文件锁**：任务目录内不写 `.lock` 文件，独占完全由 `.runtime/sessions/` 中的 session 指针决定。

**方案对比留存**（供后续参考）：

| 方案 | 机制 | 优点 | 缺点 | 状态 |
|------|------|------|------|------|
| A. 文件锁 | task start 时写 `.lock` 文件 | 强一致，简单可靠 | 同任务严格串行 | 未采用 |
| **B. session 独占指针** | 同一任务同时只能一个 session 指向 | 复用 Trellis 机制，无额外文件 | 需主动 finish 释放；需处理 stale 指针 | **已采用** |
| C. 乐观并发（OCC） | 不锁定，提交时检测冲突 | 允许并发 | AI 场景 md 合并困难 | 未采用 |
| D. 后写覆盖+告警 | 不锁定，后写覆盖先写 | 最简单 | 数据丢失风险 | 未采用 |
