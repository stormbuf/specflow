# specflow-break-loop

> 根因分析 skill。当反复调试同一类问题时激活，跳出"改了又坏"的循环。

## 核心职责

对反复出现的问题执行 5 维根因分析，判定根因，设计预防措施，将结论沉淀到 spec 库。

## 执行步骤

### 1. 描述问题现象

- 记录问题的可观察现象：输入、预期输出、实际输出
- 记录问题出现的频率与触发条件
- 列出已尝试的修复及为何无效

### 2. 5 维分析（What / Why / When / Where / How）

| 维度 | 问题 | 分析 |
|---|---|---|
| What | 到底出了什么问题？ | <精确描述异常行为> |
| Why | 为什么会出这个问题？ | <直接原因链> |
| When | 什么时候触发？ | <触发条件与时序> |
| Where | 问题定位在哪里？ | <代码位置与模块边界> |
| How | 问题是如何传播的？ | <从根因到现象的传播路径> |

### 3. 根因判定

- 区分表象与根因：表象是可观察的异常，根因是导致异常的最底层原因
- 根因判定标准：修复此点后，同类问题不再复发
- 若存在多个候选根因，逐一验证排除

### 4. 预防措施

- 针对根因设计预防措施（编码规范、测试用例、lint 规则、架构约束等）
- 预防措施须可执行、可验证
- 评估预防措施的成本与收益

### 5. 更新 spec

- 将根因分析与预防措施沉淀到 `.specflow/spec/` 对应领域
- 预防措施以规范条目形式写入，而非调试记录
- 更新 index.md 索引

## 产物

- 根因分析报告（写入任务目录或 spec 文件）
- 更新后的 spec 文件

## 约束

- 必须找到根因，不止于表象修复
- 预防措施须可执行、可验证
- spec 沉淀的是规范，不是调试日志
- 不执行 git 操作
