## 环境划分

（由团队填写：定义各部署环境的用途与配置。包括：

| 环境 | 用途 | 数据 | 访问权限 | 部署触发 |
|------|------|------|----------|----------|
| dev | 开发联调 | mock / 测试数据 | 全员 | 自动（每次 push） |
| staging | 预发布验证 | 生产数据副本（脱敏） | 限定人员 | 手动 / 自动（合并到 main） |
| prod | 生产环境 | 真实数据 | 严格限制 | 手动审批 |

包括：
- 各环境的资源配置（CPU / 内存 / 实例数）
- 环境间的数据同步策略
- 环境的访问控制与 VPN / 堡垒机
- 环境变量与配置的管理方式
）

## 部署方式

（由团队填写：定义部署的具体方式与流程。包括：

- 部署方式选型：
  - 滚动部署（Rolling Update）— 逐个替换旧实例
  - 蓝绿部署（Blue-Green）— 两套环境切换
  - 金丝雀部署（Canary）— 小流量验证后全量
  - 重建部署（Recreate）— 停旧起新
- 各方式的适用场景
- 部署窗口（是否限制部署时间）
- 部署前检查清单
- 部署后的验证步骤
- 代码示例：部署配置（如 Kubernetes manifest / Helm chart）
）

## 回滚策略

（由团队填写：定义回滚的触发条件与操作流程。包括：

- 回滚触发条件：
  - 健康检查失败
  - 错误率超过阈值
  - 核心功能不可用
  - 人工判断需要回滚
- 回滚方式：
  - 自动回滚（如 Kubernetes 的 rollout undo）
  - 手动回滚（切换镜像版本 / 切换蓝绿环境）
- 回滚的操作步骤
- 回滚后的数据处理（数据库 migration 是否需要回退）
- 回滚的演练与验证
- 代码示例：回滚命令与脚本
）

## 健康检查

（由团队填写：定义健康检查的配置与标准。包括：

- Liveness probe：判断容器是否存活，失败则重启
- Readiness probe：判断容器是否就绪，失败则不接收流量
- Startup probe：判断容器是否启动完成（慢启动场景）
- 健康检查端点的实现（如 `GET /health` / `GET /ready`）
- 健康检查的频率与超时配置
- 健康检查的内容（进程存活 / 数据库连接 / 依赖服务连通性）
- 健康检查的分层（浅检查 vs 深检查）
- 代码示例：健康检查端点实现
）

## 环境变量管理

（由团队填写：定义各环境的环境变量管理方式。包括：

- 环境变量的来源（CI secret / 配置中心 / KMS / .env）
- 各环境的环境变量清单
- 环境变量的命名规则
- 敏感变量的加密与访问控制
- 环境变量的变更流程
- 配置热更新策略（是否支持不重启更新配置）
- 代码示例：环境变量的加载与使用
）

## 常见错误

（由团队填写：列出团队在部署上犯过的错误，例如：

- 部署后不做健康检查验证，问题在用户报障时才发现
- 回滚流程从未演练，真正需要回滚时手忙脚乱
- 数据库 migration 只进不退，回滚后 schema 与代码不匹配
- 环境变量在不同环境间不同步，staging 通过但 prod 失败
- 部署无审批，任何人都能直接部署到生产
- 蓝绿部署的切换有流量丢失，用户请求中断
- 健康检查端点只返回 200，不检查依赖服务，假"健康"
- 部署配置手动修改，未纳入版本管理
- canary 部署的流量比例设置不当，小流量发现不了问题
）
