## Release Notes 格式

（由团队填写：定义 Release notes 的格式与编写流程。包括：

标准格式示例：

```markdown
# Release v1.2.0 — 2024-01-15

## 新功能
- 新增用户导出功能，支持 CSV 和 Excel 格式
- API 新增批量操作端点

## 改进
- 优化列表分页性能，大数据量下响应速度提升 50%
- 升级到 Node.js 20 LTS

## Bug 修复
- 修复夜间模式下表单文字不可见的问题
- 修复时区切换后日程显示错误的问题

## 破坏性变更
- `GET /api/users` 的 `name` 字段拆分为 `firstName` 和 `lastName`

## 依赖更新
- 升级 express 4.18.2 → 4.19.2（安全修复）
```

包括：
- Release notes 的存放位置（GitHub Release / CHANGELOG.md / Wiki）
- 编写者（自动生成 / 发布负责人）
- 编写时机（发布前 / 发布后）
- 面向受众（开发者 / 用户 / 运维）
- 是否区分内部版与外部版
）

## 发布前检查清单

（由团队填写：定义发布前必须逐项确认的检查清单。包括：

- [ ] 所有计划变更已合并到发布分支
- [ ] CI 流水线全部通过（lint / build / test / scan）
- [ ] staging 环境验证通过（功能测试 + E2E）
- [ ] 数据库 migration 已在 staging 验证
- [ ] Release notes 已编写并审查
- [ ] 版本号已更新（tag / package.json / CHANGELOG）
- [ ] 回滚方案已确认（回滚步骤可行且验证过）
- [ ] 发布窗口确认（非冻结期 / 非高峰时段）
- [ ] 通知相关方（运维 / 客服 / 产品）
- [ ] 审批已获得（如需要）
）

## 紧急修复流程

（由团队填写：定义紧急修复（hotfix）的标准流程。包括：

1. **评估** — 确认问题严重程度，判断是否需要紧急修复
2. **创建 hotfix 分支** — 基于生产版本 tag 创建
3. **修复与测试** — 最小化修改，聚焦修复问题
4. **代码审查** — 至少一人审查（紧急情况可放宽但不跳过）
5. **部署到 staging 验证** — 确认修复有效且无副作用
6. **部署到生产** — 走紧急发布流程
7. **合并回主线** — hotfix 分支合并回 main / develop
8. **事后复盘** — 记录根因与改进措施

包括：
- 紧急修复的审批流程（是否需要 CTO 级别审批）
- 紧急修复的发布窗口（是否允许任何时间）
- 紧急修复的通知方式
- 代码示例：hotfix 分支管理与合并
）

## 常见错误

（由团队填写：列出团队在发布流程上犯过的错误，例如：

- 版本号管理混乱，同一版本号对应多个不同的构建
- Release notes 只写"bug 修复和性能优化"，不写具体内容
- 发布前不检查清单，遗漏关键验证步骤
- 紧急修复不合并回主线，导致下次发布时问题复现
- 发布窗口不遵守，在高峰时段发布导致故障影响扩大
- 数据库 migration 在发布时才第一次执行，失败后卡在中间状态
- 发布后不做验证，等问题被用户发现
- 回滚方案只是"理论上可行"，从未实际演练
- 紧急修复范围蔓延，从"修一个 bug"变成"重构整个模块"
）
