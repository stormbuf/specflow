## 集成测试的范围

（由团队填写：定义集成测试覆盖的边界。包括：

- 集成测试与单元测试的分界线
- 集成测试覆盖的层组合（如 controller + service + repository）
- 是否包含真实数据库？是否包含真实消息队列？
- API 级集成测试的范围（HTTP 请求到响应的完整链路）
- 哪些依赖使用真实实现、哪些使用 fake / mock
- 集成测试与端到端测试的区别
）

## 测试环境搭建与清理

（由团队填写：定义集成测试环境的搭建与销毁方式。包括：

### 环境搭建
- 使用 docker-compose / testcontainers / 共享测试环境？
- 数据库实例的创建与 migration 执行
- 依赖服务（Redis / MQ / 对象存储）的启动
- 环境变量的配置
- 搭建时机：每个测试套件 / 每个测试 / 全局共享

### 环境清理
- 数据清理策略（truncate / drop / transaction rollback）
- 清理时机（afterEach / afterAll / 套件间）
- 确保测试间无状态残留的机制
- 资源释放（连接池 / 文件句柄 / 临时容器）
）

## 端到端测试边界

（由团队填写：定义端到端（E2E）测试的范围与执行策略。包括：

- E2E 测试覆盖的完整链路（如 前端 → API → Service → DB → 返回）
- E2E 测试的运行环境（staging / 专用 E2E 环境）
- E2E 测试的数据管理（测试账号 / 测试数据工厂）
- E2E 测试的执行频率（每次提交 / 每日 / 发版前）
- E2E 测试的稳定性保障（重试机制 / 等待策略）
- E2E 测试与集成测试的区分标准
）

## CI 中的测试策略

（由团队填写：定义测试在 CI 流水线中的执行方式。包括：

### 测试分层执行
- 哪些测试在 PR 检查时运行（通常是单元测试 + 快速集成测试）
- 哪些测试在合并后 / 定时任务中运行（通常是慢集成测试 + E2E）
- 哪些测试在发版前运行

### 并行化
- 测试是否并行执行？
- 并行分片策略（按文件 / 按套件）
- 如何避免并行测试间的资源竞争

### 速度优化
- 测试缓存策略（依赖缓存 / 构建缓存）
- 测试超时设置
- 失败快照与日志收集
- 如何标记慢测试并选择性跳过
）

## 常见错误

（由团队填写：列出团队在集成测试上犯过的错误，例如：

- 集成测试依赖共享环境，多个 PR 同时跑导致互相干扰
- 集成测试不清理数据，运行次数越多越慢、越容易失败
- E2E 测试太脆弱，UI 微调就全挂，团队开始忽略 E2E 失败
- 集成测试无超时设置，某个依赖卡住导致整个 CI 挂起
- 集成测试和单元测试混在一起，无法单独快速运行
- 集成测试使用生产数据库的副本但未脱敏，存在安全风险
- 测试数据依赖特定顺序的 migration，新环境搭建失败
- CI 中集成测试跑太久，开发者开始跳过 CI 直接合并
）
