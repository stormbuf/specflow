## 何时写测试

以下决策流程图帮助判断"这段代码是否需要写测试"：

```
                        开始
                         |
                         v
              +-------------------+
              | 这是业务逻辑吗？   |
              +-------------------+
                  /          \
               是 /            \ 否
                /              \
               v                v
        +-------------+   +-------------------+
        | 写单元测试   |   | 这是配置/常量吗？ |
        +-------------+   +-------------------+
                              /          \
                           是 /            \ 否
                            /              \
                           v                v
                    +-----------+   +---------------------+
                    | 不需要测试 |   | 这是纯 UI 展示组件？ |
                    +-----------+   +---------------------+
                                      /          \
                                   是 /            \ 否
                                    /              \
                                   v                v
                            +-----------+   +------------------+
                            | 视情况而定 |   | 这是工具函数？    |
                            | (快照测试) |   +------------------+
                            +-----------+        /          \
                                              是 /            \ 否
                                               /              \
                                              v                v
                                       +-----------+   +-------------------+
                                       | 写单元测试 |   | 写集成测试         |
                                       +-----------+   +-------------------+
```

### 必须写测试的场景

- 业务逻辑 / 领域模型（核心规则、状态机、计算）
- 工具函数 / 纯函数（输入输出可预测）
- 数据转换 / 序列化反序列化
- API 端点（至少集成测试覆盖 happy path）
- 复杂的分支逻辑

### 可以不写测试的场景

- 纯配置文件 / 常量定义
- 纯类型声明（无运行时逻辑）
- 纯展示组件（无交互逻辑、无状态）
- 第三方库的包装（如果只是透传，测库自身的行为没有意义）
- 一次性脚本 / 临时代码

### 测试隔离策略

（由团队填写：定义测试之间的隔离机制。包括：

- 文件系统隔离（每个测试使用独立 tmpdir）
- 环境变量隔离（测试前备份、测试后恢复）
- 数据库隔离（每个测试独立事务 / 独立数据库 / truncate）
- 网络隔离（禁止真实网络请求 / 使用 mock server）
- 全局状态清理（afterEach 重置全局状态）
- 测试执行顺序（是否依赖执行顺序、是否随机执行）
）

## 常见错误

（由团队填写：列出团队在测试约定上犯过的错误，例如：

- 测试依赖执行顺序，单独跑通过、全量跑失败
- 测试共享数据库状态，互相污染导致随机失败
- 测试断言不充分，只验证"不报错"不验证"结果正确"
- 测试名描述"应该正常工作"，无法从名称看出验证了什么
- 把 UI 快照测试当单元测试，每次样式微调都全部失败
- 测试中硬编码日期 / 时间，在特定时间点后失败
- mock 了太多东西，测试实际上什么都没测到
- 覆盖率达标但测试无断言，只是"跑过"
）
