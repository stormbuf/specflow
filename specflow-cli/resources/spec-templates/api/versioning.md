## 向后兼容策略

（由团队填写：定义什么变更算 breaking change 以及兼容性规则。包括：

### 非 breaking change（不需要新版本）
- 新增可选请求字段
- 新增响应字段
- 新增 API 端点
- 放宽输入验证规则
- 改善错误信息描述

### Breaking change（需要新版本）
- 删除或重命名已有字段
- 改变字段类型
- 收紧输入验证规则
- 改变 HTTP 方法语义
- 改变错误响应格式
- 改变默认行为
- 改变分页 / 排序的默认值

（由团队填写：补充团队实际的兼容性判定标准与处理流程。）

## 废弃 API 流程

（由团队填写：定义废弃旧版 API 的标准流程。包括：

1. **标记废弃** — 在 API 文档和响应 Header 中标记 `Deprecation` 和 `Sunset`
2. **通知期** — 保持旧版可用，设定期望淘汰日期（如 6 个月）
3. **迁移支持** — 提供迁移指南，说明新版的变化与适配方法
4. **监控使用** — 监控旧版的调用量，联系仍在使用的关键客户
5. **下线** — 到期后返回 410 Gone 或重定向到新版
6. **记录** — 在变更日志中记录下线信息

（由团队填写：补充团队实际的废弃周期与通知方式。）
）

## 变更日志格式

（由团队填写：定义 API 变更日志的格式与维护方式。包括：

标准格式示例：

```markdown
## [v1.2.0] - 2024-01-15

### Added
- 新增 `GET /users/:id/orders` 端点，查询用户订单列表
- 响应新增 `last_login_at` 字段

### Changed
- `POST /users` 的 `email` 字段验证规则收紧，需符合 RFC 5322

### Deprecated
- `GET /users?fields=all` 参数废弃，请使用 `GET /users?expand=profile`
- `user.status` 字段中的 `0` 值废弃，请使用 `inactive`

### Fixed
- 修复分页 cursor 在特定条件下重复返回数据的问题

### Removed
- 移除 `GET /v1/users` 中的 `legacy_id` 字段（已于 v1.0 废弃）
```

包括：
- 变更日志的存放位置（CHANGELOG.md / API 文档 / Wiki）
- 变更分类标准（Added / Changed / Deprecated / Fixed / Removed）
- 版本号规则（与 SemVer 的关系）
- 变更日志的审查与发布流程
）

## 常见错误

（由团队填写：列出团队在 API 版本管理上犯过的错误，例如：

- 没有 API 文档或文档与实现不同步
- 直接修改已有 API 字段类型，导致客户端崩溃
- 废弃 API 不通知就直接下线，客户端突然收到 404
- 变更日志只写"修复了一些 bug"，不写具体内容
- 同时维护太多版本，维护成本失控
- 新版本不提供迁移指南，使用者需要自行对比差异
- 版本号管理混乱，v1.1 和 v1.2 之间有 breaking change 但未升级大版本
- 在非 breaking change 时也创建新版本，导致版本碎片化
- 响应中突然新增必填字段，导致旧客户端解析失败
）
