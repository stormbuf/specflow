# 架构决策记录

本目录记录项目中重要的架构决策及其上下文、决策与后果（ADR, Architecture Decision Records）。

## 说明

架构决策记录（ADR）用于沉淀项目中那些"做出选择后难以轻易回退"的技术决策。每条 ADR 记录决策时的完整上下文，使未来的人（包括 AI）能理解"为什么当初这么选"，而不是只看到结果。

ADR 一旦写入即不再修改正文；如果决策被推翻或废弃，新建一条 ADR 标记其状态变更，并在原 ADR 中添加指向新 ADR 的链接。

## ADR 编写说明

1. **何时写 ADR** — 当一个决策满足以下任一条件时写：影响多个模块、引入或移除主要依赖、改变数据流或部署拓扑、难以回退。日常的"变量怎么命名"不写 ADR。

2. **文件命名** — `adr-<编号>-<kebab-case-标题>.md`，如 `adr-001-use-postgresql.md`。编号从 001 开始递增，不复用。

3. **编写流程** — 复制 [adr-template.md](adr-template.md) 为新文件，填写各章节，然后在本表格追加一行。决策日期填写提交日期。

4. **状态管理** — ADR 状态变更（如废弃）时，不要删除原文件，只改状态字段并在底部追加"状态变更记录"指向新 ADR。

5. **内容原则** — 记录决策时的真实上下文与权衡，不要倒果为因（已知结果再编理由）。替代方案章节尤其重要，它说明"为什么不选其他路"。
