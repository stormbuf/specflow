# RULES.md — 全流程约束

> 本文件供 任务拆解和执行阶段注入检查项。
> 改频率低，改了下次拆解/执行自动生效。

---

## §1 思考先行

写代码前先列改动文件清单 + 意图。不在不确定时直接动手。

## §2 小步迭代

每次只改一个任务。不改功能+重构混合。任务间的改动不应交叉。

## §3 契约显式化

- 对外接口、事件、配置、数据结构的语义必须显式。
- 状态值、枚举值、错误语义和边界行为必须有单一真相源。
- 跨模块共享契约必须集中定义，避免多个模块重复维护同一语义。
- 实现必须遵循 design.md 中已确认的接口、数据流、状态流转、UML 图和约束。

## §4 单一职责

- 每个模块/函数职责可用一句话描述，且描述不含"和"字
- 不为单次使用创建抽象
- 不引入未要求的"灵活性"或"可配置性"

## §5 怀疑就停

遇到模糊点（需求不清、技术判断不确定、PRD 与 design 有歧义）立即暂停，列出待澄清项。不自行假设。

## §6 沟通原则

- 原子修改（<20 行，单文件）事后摘要告知
- 中等及以上变更（多文件 / 改对外接口 / 改数据结构或持久化结构）先出方案等确认

## §7 影响面扫描

改动前做影响面扫描，列出受影响文件、接口、数据、配置或流程。确认影响面可控后再开始改。

## §8 错误处理策略

- 所有外部输入、外部依赖、IO、网络、并发、状态转换和持久化操作必须考虑失败路径。
- 错误语义必须显式：哪些错误可恢复、哪些错误应阻断、哪些错误需要上抛或转换。
- 对外接口返回的错误结构、错误码、错误消息或失败状态必须稳定且可验证。
- 不吞掉错误；确需降级时必须记录降级条件、用户可见影响和后续恢复方式。
- 错误日志不得泄露密钥、令牌、隐私数据或内部敏感实现细节。
- 错误处理必须有对应验证：测试、复现步骤或人工检查项。

## §9 测试策略

### §9.1 单元测试（始终触发）

涉及业务逻辑、状态判断、数据转换、边界处理或错误处理时，必须添加或更新单元测试。外部依赖使用 mock 的方式处理。

### §9.2 属性测试（条件触发）

触发条件：函数涉及数据转换 或 存在数学不变量 或 处理集合操作。

不触发时标注 `[SKIP]` 并说明原因。

### §9.3 集成测试（条件触发）

触发条件：新增或修改对外接口、数据结构、配置契约、跨模块数据流转，或动了关键用户旅程。

不触发时标注 `[SKIP]` 并说明原因。

### §9.4 回归测试（条件触发）

触发条件：修改了已有代码逻辑且存在存量功能。

不触发时标注 `[SKIP]` 并说明原因（如"新功能，无存量行为需保护"）。

### §9.5 冒烟测试（条件触发）

**定位**：发布或交付前对真实运行路径的快速连通性验证。与 §9.3 集成测试的关键区别：冒烟测试使用真实依赖或目标环境，不使用 mock。

触发条件（满足任一即适用，必须逐条判断）：

- 本次变更涉及外部依赖接入
- 存在可用的发布、部署或交付流程
- 新增或修改了环境变量 / 配置文件

执行时机：发布、部署或交付准备完成后执行，在 mock-based 测试通过之后。

质量标准：

- 从关键旅程中选取 3~5 条端到端路径
- 禁止 mock：关键外部调用必须走真实依赖或目标环境
- 总执行时间 ≤ 120 秒
- 任一条失败 → 阻断发布或交付，并按项目回退策略处理

不触发时标注 `[SKIP]` 并说明原因。

## §10 安全扫描（条件触发）

触发条件：引入新外部依赖，修改外部输入处理，新增对外入口，或动了认证/授权逻辑。

不触发时标注 `[SKIP]` 并说明原因。

扫描项：

- 密钥、令牌或凭据是否经日志、错误信息或响应输出
- 环境、配置或密钥路径是否写死在代码中
- 外部输入是否进入命令执行、文件路径、查询语句或模板渲染
- 数据目录是否受外部输入控制

## §11 执行策略

- 如果平台支持多 agent 机制，由调度 agent 根据任务区分并调度代码生成 agent、测试编写与执行 agent、代码审查 agent。
- 功能按 tasks.md 分组串行执行；同一功能内按可独立验证子项顺序推进。
- 每个可独立验证子项按“代码生成 -> 测试编写与执行 -> 代码审查”的顺序推进。
- 一个功能的所有子项完成后，再执行该功能级质量门禁。
- 测试或审查暴露的问题交由代码生成 agent 修复；修复后重新进入测试和审查。
- 同一任务最多返修三轮；三轮后仍有问题时，阻断执行并告知用户问题、已尝试修复和下一步选择。
- 如果平台不支持多 agent 机制，则由当前 agent 按相同顺序串行执行代码生成、测试编写与执行、代码审查和返修。

## §12 代码审查

apply 执行阶段必须完成代码审查。

审查项：

- 需求一致性：实现是否覆盖 proposal、spec-delta、design 和 tasks 中的要求。
- 设计一致性：实现是否遵循 design.md 中确认的接口、数据流、状态流转、UML 图和约束。
- 任务完整性：tasks.md 中对应任务是否完成，是否存在遗漏项或未说明例外。
- 变更边界：是否修改了无关文件、无关逻辑或顺手重构。
- 契约完整性：对外接口、数据结构、配置、错误语义和边界行为是否清晰。
- 测试覆盖：适用的单元级验证、属性测试、集成测试、回归测试是否已添加或说明不适用原因。
- 安全风险：是否引入凭据泄露、外部输入未约束、权限绕过、路径/命令/查询注入等风险。
- 可维护性：实现是否过度抽象、重复逻辑、职责混杂或引入未要求的灵活性。

## §13 质量门禁

每个功能完成后运行项目已定义的适用检查：

- 类型或静态检查
- 静态分析、格式或风格检查
- 自动化测试
- 构建或打包检查

若项目没有明确定义对应命令，记录“未发现可执行命令”，不得臆造命令。

## §14 执行纪律

- 禁止在未获确认的情况下继续执行
- 禁止自行假设需求、补充需求
- 禁止顺便改无关代码（如格式化相邻行、重构未涉及的函数）
