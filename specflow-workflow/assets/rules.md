# RULES.md

> 本文件定义检查约束和编码原则，供 Tasks 和 Apply 阶段动态消费。
> 改动后下次执行自动生效。

---

## 检查项

> Tasks 阶段读取本分类中的规则，将适用的编排为任务项。
> Apply 阶段执行对应任务，并最终校验覆盖。

### 测试策略

| 测试类型 | 粒度   | 时机                   | 动作             |
| -------- | ------ | ---------------------- | ---------------- |
| 单元测试 | 子项级 | 子项实现后、审查前     | 编写             |
| 属性测试 | 子项级 | 子项单测通过后、审查前 | 编写             |
| 集成测试 | 功能级 | 全部子项完成后、门禁前 | 编写或更新       |
| 回归测试 | 功能级 | 质量门禁中             | 运行已有         |
| 冒烟测试 | 交付级 | 门禁通过后、交付前     | 运行（真实依赖） |

#### 单元测试

> 粒度：子项级  /  时机：子项实现后 → 审查前

- **Given** 子项涉及业务逻辑、状态判断、数据转换、边界处理或错误处理
- **When** 该子项的实现代码完成
- **Then** 编写或更新单元测试，外部依赖使用 mock。执行并确认当前子项的单测全部通过后才可进入审查

不适用时标注 `[SKIP]` 并说明原因。

#### 属性测试

> 粒度：子项级  /  时机：子项单测通过后 → 审查前

- **Given** 子项涉及数据转换、数学不变量或集合操作
- **When** 该子项的单测全部通过
- **Then** 编写属性测试（如随机输入验证不变量恒成立）。确认当前子项的属性测试全部通过后才可进入审查

不触发时标注 `[SKIP]` 并说明原因。

#### 集成测试

> 粒度：功能级  /  时机：全部子项完成后 → 门禁前

- **Given** 本次变更新增或修改了对外接口、数据结构、配置契约、跨模块数据流转，或动了关键用户旅程
- **When** 该功能的所有子项均已完成
- **Then** 编写或更新集成测试，覆盖跨模块交互路径和契约正确性。确认当前功能的集成测试全部通过后才可进入质量门禁

不触发时标注 `[SKIP]` 并说明原因。

#### 回归测试

> 粒度：功能级  /  时机：质量门禁中

- **Given** 本次变更修改了已有代码逻辑，且项目存在已有测试
- **When** 该功能所有子项和集成测试完成后，进入质量门禁
- **Then** 运行全部已有测试，确认无回归。任一失败需修复后重跑，全部已有测试通过方可标记门禁通过

不触发时标注 `[SKIP]` 并说明原因（如"新功能，无存量行为需保护"）。

#### 冒烟测试

> 粒度：交付级  /  时机：门禁通过后 → 交付前

- **Given** 本次变更涉及外部依赖接入，或存在可用发布流程，或新增/修改了环境变量或配置文件
- **When** 质量门禁全部通过，交付或发布准备就绪
- **Then** 从关键用户旅程中选取 3~5 条端到端路径，使用真实依赖执行（禁止 mock）。总耗时 ≤ 120 秒。任一条失败则阻断交付

不触发时标注 `[SKIP]` 并说明原因。如项目尚无交付流程，记录"待部署后执行"。

### 安全扫描

> 粒度：子项级  /  时机：子项代码完成后

- **Given** 本次变更引入新外部依赖，或修改外部输入处理，或新增对外入口，或动了认证/授权逻辑
- **When** 该子项的代码编写完成
- **Then** 扫描以下项，发现问题修复后再进入测试：
  - 密钥、令牌或凭据是否经日志、错误信息或响应输出
  - 环境、配置或密钥路径是否写死在代码中
  - 外部输入是否进入命令执行、文件路径、查询语句或模板渲染
  - 数据目录是否受外部输入控制

不触发时标注 `[SKIP]` 并说明原因。

### 代码审查

> 粒度：子项级  /  时机：子项测试通过后

- **Given** 每个子项（始终适用）
- **When** 该子项的测试全部通过
- **Then** 按以下条目逐项审查，发现问题修复后再进入下一子项或门禁：
  - 需求一致性：实现是否覆盖 proposal、spec-delta、design 和 tasks 中的要求
  - 设计一致性：实现是否遵循 design.md 中确认的接口、数据流、状态流转、UML 图和约束
  - 任务完整性：tasks.md 中对应任务是否完成，是否存在遗漏项或未说明例外
  - 变更边界：是否修改了无关文件、无关逻辑或顺手重构
  - 契约完整性：对外接口、数据结构、配置、错误语义和边界行为是否清晰
  - 测试覆盖：适用的单元测试、属性测试、集成测试、回归测试是否已添加或说明不适用原因
  - 安全风险：是否引入凭据泄露、外部输入未约束、权限绕过、路径/命令/查询注入等风险
  - 可维护性：实现是否过度抽象、重复逻辑、职责混杂或引入未要求的灵活性

### 质量门禁

> 粒度：功能级  /  时机：全部子项通过后

- **Given** 每个功能（始终适用）
- **When** 该功能的所有子项、测试、安全扫描和审查均已完成
- **Then** 运行项目已定义的适用检查：
  - 类型或静态检查
  - 静态分析、格式或风格检查
  - 自动化测试
  - 构建或打包检查

若项目没有明确定义对应命令，记录"未发现可执行命令"，不得臆造命令。

### 多 Agent 策略

> 粒度：子项级  /  时机：Apply 执行期间
>
> 本分类为执行策略，不是检查项。Tasks 阶段不在本分类上编排任务项（无 Given/When/Then 结构，自然跳过）；Apply 阶段读取本分类作为 agent 调度依据。

#### Agent 角色

| 角色           | 适用任务      | 触发场景                               |
| -------------- | ------------- | -------------------------------------- |
| 代码生成 agent | 功能实现      | 所有非测试代码的编写和修改             |
| 测试 agent     | 测试编写/更新 | 单元测试、属性测试、集成测试（功能级） |
| 审查 agent     | 代码审查      | 每次子项实现+测试完成后                |
| UI agent       | UI/UX 实现    | 涉及界面或交互变更                     |
| 调研 agent     | 文档/代码搜索 | 需查阅外部文档或搜索代码模式           |
| 编排器         | 安全扫描      | 代码生成完成后、测试前执行（编排器自行执行） |

#### 子项内串行规则

同一子项的三个阶段必须串行，不可并行：

1. 代码生成 agent（或编排器）→ 实现
2. 测试 agent（或编排器）→ 测试编写/更新 → 执行验证 → 未通过时：
   - 测试代码问题（断言、mock、测试逻辑错误）→ 测试 agent 修复
   - 非测试代码问题（实现 bug）→ 退回代码生成 agent 修复 → 重新执行测试
   - 修复后重跑，最多三轮；三轮内全部通过即完成，三轮后仍有失败则阻断
3. 审查 agent（或编排器）→ 代码审查
   - 审查通过 → 子项完成
   - 审查发现问题 → 退回代码生成 agent 修复 → 重新执行测试验证 → 重新审查
   - 最多三轮；三轮后仍有问题则阻断

审查退回后的修复循环中，测试失败的分流规则与步骤 2 相同（测试代码问题→测试 agent，实现问题→代码生成 agent），轮次计入审查阶段上限，不额外消耗测试阶段额度。

前一阶段完成后才能进入下一阶段。测试阶段必须以全部测试通过为完成标志。

#### 降级兼容

若提示词环境不支持多 agent 委托，编排器按上述串行规则自行执行全部三步，不阻塞流程，不跳过阶段。

#### 并行规则

- 不同子项如无依赖关系，可并行调度
- 调研任务可与实现并行启动
- 子项内部的实现→测试→审查不可并行

#### 不委托的情况

- 解释开销大于直接执行
- 编排器自行执行更高效

## 编码原则

> 不在 tasks.md 中生成任务项。Apply 执行时作为脑内规则实时遵循。

### 思考先行

写代码前先列改动文件清单 + 意图。不在不确定时直接动手。

### 契约显式化

- 对外接口、事件、配置、数据结构的语义必须显式。
- 状态值、枚举值、错误语义和边界行为必须有单一真相源。
- 跨模块共享契约必须集中定义，避免多个模块重复维护同一语义。
- 实现必须遵循 design.md 中已确认的接口、数据流、状态流转、UML 图和约束。

### 单一职责

- 每个模块/函数职责可用一句话描述，且描述不含"和"字
- 不为单次使用创建抽象
- 不引入未要求的"灵活性"或"可配置性"

### 影响面扫描

改动前做影响面扫描，列出受影响文件、接口、数据、配置或流程。确认影响面可控后再开始改。

### 错误处理策略

- 所有外部输入、外部依赖、IO、网络、并发、状态转换和持久化操作必须考虑失败路径。
- 错误语义必须显式：哪些错误可恢复、哪些错误应阻断、哪些错误需要上抛或转换。
- 对外接口返回的错误结构、错误码、错误消息或失败状态必须稳定且可验证。
- 不吞掉错误；确需降级时必须记录降级条件、用户可见影响和后续恢复方式。
- 错误日志不得泄露密钥、令牌、隐私数据或内部敏感实现细节。
- 错误处理必须有对应验证：测试、复现步骤或人工检查项。
