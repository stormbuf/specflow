## 上游产物依赖表（Tasks 阶段查表入口）

| 变更类型 | proposal | spec-delta | design | architecture | 验证语义 | 子项拆解依据 |
|---|---|---|---|---|---|---|
| 功能族（functional） | 必需 | 必需 | 必需 | 必需（若生代码） | Gherkin 验收全绿 | design §1（模块定位）+ §4（接口设计）；每个对外接口/独立模块 = 一个子项 |
| 优化族（nonfunctional） | 必需 | 不适用 | 必需 | 视情况 | 指标达标 + 行为回归 | design §2（优化方案设计）的各优化点；每个独立优化点 = 一个子项 |
| 迁移族（infrastructure） | 必需 | 不适用 | 必需 | 必需 | 迁移完成 + 行为回归 | design §2（迁移流程设计）的各阶段 + §3（数据迁移设计）的各迁移点；每个迁移阶段/受影响模块 = 一个子项 |
| 轻量族（lightweight） | 必需 | 不适用 | 不适用 | 不适用 | 手动核对 | proposal 范围直接映射为单一任务 |

**变更类型归属规则：**

- 功能族 = 有行为变更 + 有验收标准（spec-delta 存在）
- 优化族 = 无行为变更 + 有内部优化目标（性能 / 安全 / 重构 / 技术债 / 可观测性 / 可靠性）
- 迁移族 = 无行为变更 + 有结构迁移（升级 / 迁移 / CI / 容器化 / 配置变更）
- 轻量族 = 极小变更（文档 / 测试补充 / 合规 / License / 文案 / 样式）
- 重构归入优化族（无行为变更 + 内部结构改进，共享"基线 → 实现 → 对比"拓扑）
- bugfix：有行为变更 → 功能族；极小修复 → 轻量族
- 依赖升级：patch 级 → 轻量族；major/breaking → 迁移族

## 实现 section — 子项组织原则（所有族通用）

- 同一子项内的步骤严格串行，顺序由各族拓扑定义。
- 不同子项若无依赖关系，可并行委托。
- 调研任务可与实现并行。
- 委托决策按下方 Agent 角色表；不委托的情况：解释开销 > 直接执行 / 编排器自行更高效。

### Agent 角色

| 角色 | 适用任务 |
|---|---|
| 代码生成 agent | 所有非测试代码的编写和修改 |
| 测试 agent | 单测/属性/集成测试编写与执行 |
| 审查 agent | 子项实现+测试完成后的代码审查 |
| UI agent | 涉及界面或交互的变更 |
| 调研 agent | 查阅外部文档或搜索代码模式 |
| 安全扫描 agent | 子项涉及外部输入/依赖/认证授权变更 |

**降级：** 按 agent 粒度降级——每个任务项对应 agent 可用则委托，不可用则编排器自行执行。不因单个 agent 不可用而全局降级。

## ▼ 任务族插槽（四选一，整块装配）

### 任务族 A：功能族（functional）

> 子项拆解依据：design §1（模块定位）+ §4（接口设计）。每个对外接口 / 独立模块 = 一个子项。
> 每个子项**强制 `covers:` 字段**标注覆盖的 spec-delta Gherkin Scenario。
> 子项内 5 步，测试骨架先行：

```
#### <N.M> <子项名称>
- [ ] <N.M.1> 测试骨架 — agent: 测试 | covers: spec-delta §<需求名>.<Scenario 名>
- [ ] <N.M.2> 实现 — agent: 代码生成
- [ ] <N.M.3> 安全扫描 — agent: 安全扫描
- [ ] <N.M.4> 单测/属性测试 — agent: 测试
- [ ] <N.M.5> 审查 — agent: 审查
```

**测试骨架说明：** 从 spec-delta Gherkin Scenario 派生验收用例骨架，固定覆盖范围（函数签名、输入输出类型、关键场景、边界条件、错误路径）。不要求测试可运行，但必须先于实现产出。实现以测试骨架为通过标准。

**集成测试（功能级，所有子项完成后）：**

- [ ] <N+1>.1 集成测试：覆盖跨模块交互路径和契约正确性 — agent: 测试

---

### 任务族 B：优化族（nonfunctional）

> 子项拆解依据：design §2（优化方案设计）的各优化点。每个独立优化点 = 一个子项。
> 每个子项 `covers:` 标注 design §2 优化点。
> 子项内 3 步，对比驱动：

```
#### <N.M> <优化点名称>
- [ ] <N.M.1> 基线测量：记录当前指标（性能 / 行为回归基线） — covers: design §1b.<优化点>
- [ ] <N.M.2> 实现 — agent: 代码生成
- [ ] <N.M.3> 优化后测量对比 — agent: 测试
```

**整体审查（所有子项后）：**

- [ ] <N+1>.1 代码审查：整体审查 — agent: 审查

**专项验证：**

- [ ] <N+2>.1 回归测试：运行已有测试确认无回归
- [ ] <N+2>.2 性能对比 / 安全扫描 / 故障注入（按优化类型选）

---

### 任务族 C：迁移族（infrastructure）

> 子项拆解依据：design §2（迁移流程设计）的各阶段 + §3（数据迁移设计）的各迁移点。每个迁移阶段 / 受影响模块 = 一个子项。
> 每个子项 `covers:` 标注 design §2/§3 迁移点。
> 子项内 2 步，迁移驱动：

```
#### <N.M> <迁移单元名称>
- [ ] <N.M.1> 迁移单元实现（适配代码 / 迁移脚本 / 配置更新） — agent: 代码生成 | covers: design §2/§3.<迁移点>
- [ ] <N.M.2> 迁移单元验证（数据一致性 / 服务连通性 / 功能可用性）
```

**整体审查（所有子项后）：**

- [ ] <N+1>.1 代码审查：整体审查 — agent: 审查

**迁移/切换验证：**

- [ ] <N+2>.1 数据一致性验证
- [ ] <N+2>.2 服务连通性验证
- [ ] <N+2>.3 功能可用性验证

**回归 + 基线对比：**

- [ ] <N+3>.1 回归测试：运行已有测试确认无回归
- [ ] <N+3>.2 基线对比：对比关键指标

## Section 启用条件清单

各 section 的启用条件（替代原 rules.md 的 Given/When/Then）。满足任一即启用，否则标 `[SKIP]` 并写明未命中哪条。

### 单元测试（子项级）

- **启用条件**：子项涉及业务逻辑、状态判断、数据转换、边界处理或错误处理。
- **时机**：子项实现后、审查前。外部依赖使用 mock。

### 属性测试（子项级）

- **启用条件**：子项涉及数据转换、数学不变量或集合操作。
- **时机**：子项单测通过后、审查前。

### 集成测试（功能级）

- **启用条件**：本次变更新增或修改了对外接口、数据结构、配置契约、跨模块数据流转，或动了关键用户旅程。
- **时机**：全部子项完成后、门禁前。执行验证由质量门禁中的全量测试统一完成。

### 回归测试（功能级）

- **启用条件**：本次变更修改了已有代码逻辑，且项目存在已有测试。
- **时机**：质量门禁中。运行全部已有测试，任一失败修复后重跑。

### 冒烟测试（交付级）

- **启用条件**：本次变更涉及外部依赖接入，或存在可用发布流程，或新增/修改了环境变量或配置文件。
- **时机**：门禁通过后、交付前。从关键用户旅程选取 3~5 条端到端路径，真实依赖执行（禁止 mock），总耗时 ≤ 120 秒。无交付流程时记录"待部署后执行"。

### 安全扫描（子项级）

- **启用条件**（满足任一）：
  - 子项引入新的外部依赖
  - 子项修改了外部输入处理逻辑
  - 子项新增对外入口（接口 / 路由 / CLI 命令）
  - 子项涉及认证或授权变更
- **扫描项**：密钥/令牌/凭据泄露、环境/配置/密钥路径硬编码、外部输入进入命令执行/文件路径/查询语句/模板渲染、数据目录受外部输入控制。
- **时机**：子项代码完成后，发现问题修复后再进入测试。

### 代码审查（子项级）

- **启用条件**：始终适用（每个子项）。
- **时机**：子项测试全部通过后。逐项审查：需求一致性、设计一致性、任务完整性、变更边界、契约完整性、测试覆盖、安全风险、可维护性。发现问题修复后再进入下一子项或门禁。

### 质量门禁（功能级）

- **启用条件**：始终适用。
- **检查项**：类型或静态检查、静态分析/格式/风格检查、自动化测试、构建或打包检查。无对应命令时记录"未发现可执行命令"。
