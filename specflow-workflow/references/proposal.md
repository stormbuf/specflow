# Proposal Stage

本文件定义提议讨论阶段。用于把用户意图收敛为一次可执行变更的边界，不写实现细节。

路径锚定：本文件中的 `specflow/`、源码、测试和项目文档路径均相对于 `{PROJECT_ROOT}/`；已有 change 只在 `{PROJECT_ROOT}/specflow/changes/<change-id>/` 下查找。

## 目标

- 明确为什么做、解决什么问题、为什么现在做。
- 明确范围和非范围。
- 判断变更类型、是否需要规约、是否需要技术方案。
- 记录开放问题；关键问题未解决时不得进入后续阶段。

## 输入

- 用户原始需求；`specflow/roadmap.md`，如果存在待实现项。
- 已有 `specflow/changes/<change-id>/` 内容，如存在。
- 相关长期文档：`specflow/specs/`、`specflow/architecture.md`、`specflow/adr/`，仅在判断影响时读取。

## Roadmap 入口

```text
IF specflow/roadmap.md 存在且包含 📋 下一批、💡 远期 或 🔥 正在进行 中未完成条目:
  询问用户是否从 roadmap 选择本次执行项
IF 用户选择条目: 在 proposal.md 的 Roadmap 来源中记录所选 F/T 条目
ELSE:
  Roadmap 来源写“无”
```

选择多个 roadmap 条目时，必须确认它们属于同一次可批准、可设计、可执行、可验证、可归档的变更边界；否则提示拆成多次 proposal。

## 事实边界

Proposal 只能基于以下事实来源生成：

- 用户原始需求中已经明确表达的目标、问题、范围或限制。
- 当前对话中用户已经确认的上下文。
- 已有 `specflow/specs/`、`specflow/architecture.md`、`specflow/adr/` 中的长期项目事实。
- 已有源码、测试或项目文档中可验证的当前系统事实。

不得静默补全用户没有表达、项目事实也无法证明的业务动机、范围、非范围或影响面。缺失的关键事实必须写入开放问题，并使用苏格拉底追问询问用户。

```text
IF proposal 字段需要补充事实才能成立:
  先检查用户原话、当前对话、主 spec、架构文档、ADR、源码、测试或项目文档是否已有依据
ELSE IF 没有事实依据:
  将缺失事实写入开放问题
  使用苏格拉底追问询问用户并暂停
ELSE:
  基于已确认事实填写 proposal
```

## 字段生成规则

- 背景与原因：只写用户明确表达的问题、目标、痛点或已确认背景；缺失时追问。
- 范围 / 包含：只写用户明确要求做的内容，或项目事实能证明属于本次变更的内容。
- 范围 / 不包含：只写用户明确排除的内容；不得为了完整性替用户排除功能。
- 变更类型：AI 可以基于变更性质判断，但必须说明理由。
- 是否需要规约：AI 可以基于是否改变可观察行为判断，但必须引用依据。
- 是否需要技术方案：AI 可以基于影响面和风险判断，但必须引用依据。
- 目标能力：只能基于用户明确能力、已有 spec 名称或可验证项目事实；不确定时写入开放问题。
- 影响面：只写已知或可从项目结构确认的影响；不确定时标注未知或写入开放问题。
- 开放问题：收纳无法从事实来源确认、且会影响范围、规约、设计或执行的关键问题。
- Roadmap 来源：只记录用户选择的 roadmap F/T 条目；不从 roadmap 静默扩展范围。

## 苏格拉底追问

追问目标是澄清变更边界，而不是替用户决定业务方向。

```text
IF 缺失事实影响范围、是否需要规约、是否需要技术方案、目标能力或影响面:
  先指出当前无法确定的事实
  再说明该事实为什么影响后续阶段
  最后提出 1-3 个聚焦问题
ELSE:
  不追问，继续整理已确认意图
```

问题应遵守：

- 不诱导用户接受 AI 的默认方案。
- 不一次性抛出大量选项。
- 优先询问会改变范围、是否需要规约或是否需要技术方案的事实。
- 如果用户回答仍然模糊，继续追问边界条件或反例。

示例：

```text
当前需求只说明“增加导出能力”，但没有说明导出对象、格式和权限边界。
这些信息会影响范围、是否需要规约、目标能力和后续 Spec Delta。

1. 本次导出能力面向普通用户、管理员，还是两者都包含？
2. 导出格式是否已经确定？
3. 是否有明确不做的范围，例如异步导出或批量导出？
```

## 输出

- `specflow/changes/<change-id>/proposal.md`

## 变更类型

- `tiny`：文案、样式、低风险等价替换。
- `functional`：改变用户或系统可观察行为。
- `bugfix`：修复已有行为偏差。
- `infrastructure`：构建、依赖、CI、运行环境、部署。
- `architecture`：系统边界、模块职责、数据流、安全模型等长期结构。
- `foundation`：初始约定、目录、术语、质量门禁、初始架构。
- `documentation`：文档、规则、模板更新。

允许组合类型，但只在确实跨类别时使用。

## 是否需要规约判断

```text
IF 变更改变用户或系统可观察行为:
  是否需要规约 = yes
ELSE:
  是否需要规约 = no，并说明理由
```

可观察行为包括 UI 行为、API 响应、CLI 参数、错误语义、状态变化、数据保留策略、同步或一致性语义。

## 是否需要技术方案判断

```text
IF 变更涉及跨模块影响、API/CLI/数据模型/状态流变化、架构边界、安全、部署、外部依赖、多方案取舍、高返工风险或长期约束:
  是否需要技术方案 = yes
ELSE:
  是否需要技术方案 = no，并说明理由
```

## 阻断

```text
IF 范围 / 不包含 不清楚:
  使用苏格拉底追问询问用户并暂停
ELSE IF 是否需要规约 或 是否需要技术方案 判断缺少理由:
  补齐理由
ELSE IF 存在影响方案的开放问题:
  使用苏格拉底追问询问用户并暂停
ELSE IF proposal 包含没有事实来源的新业务动机、范围、非范围或影响面:
  暂停，移入开放问题并使用苏格拉底追问询问用户
```

## 完成条件

- `proposal.md` 存在，且 Roadmap 来源已记录；没有来源时明确写“无”。
- 范围 / 不包含 明确。
- 变更类型、是否需要规约、是否需要技术方案均有结论和理由。
- 无阻断性开放问题。
- 无未标明事实来源的新业务动机、范围、非范围或影响面。
