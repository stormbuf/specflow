# Proposal Stage

本文件定义提议讨论阶段。用于把用户意图收敛为一次可执行变更的边界，不写实现细节。

路径锚定：本文件中的 `specflow/`、源码、测试和项目文档路径均相对于 `{PROJECT_ROOT}/`；已有 change 只在 `{PROJECT_ROOT}/specflow/changes/<change-id>/` 下查找。

## 目标

- 明确为什么做、解决什么问题、为什么现在做。
- 明确范围和非范围。
- 判断变更类型、是否需要规约、是否需要技术方案。
- 记录开放问题；关键问题未解决时不得进入后续阶段。

## 输入

- 用户原始需求；`specflow/roadmap.md`，如果存在待实现项。
- 已有 `specflow/changes/<change-id>/` 内容，如存在。
- 相关长期文档：`specflow/specs/`、`specflow/architecture.md`、`specflow/adr/`，仅在判断影响时读取。

## Roadmap 入口

```text
IF specflow/roadmap.md 存在且包含 📋 下一批、💡 远期 或 🔥 正在进行 中未完成条目:
  询问用户是否从 roadmap 选择本次执行项
IF 用户选择条目:
  在 proposal.md 的 Roadmap 来源中记录所选 F/T 条目
  将所选条目从原分区移入 🔥 正在进行，追加 change-id
  若条目已在 🔥 正在进行，仅追加 change-id（不重复移动）
ELSE:
  Roadmap 来源写"无"
```

选择多个 roadmap 条目时，必须确认它们属于同一次可批准、可设计、可执行、可验证、可归档的变更边界；否则提示拆成多次 proposal。

## 事实边界

Proposal 只能基于以下事实来源生成：

- 用户原始需求中已经明确表达的目标、问题、范围或限制。
- 当前对话中用户已经确认的上下文。
- 已有 `specflow/specs/`、`specflow/architecture.md`、`specflow/adr/` 中的长期项目事实。
- 已有源码、测试或项目文档中可验证的当前系统事实。

若项目尚未建立 specflow/specs/、architecture.md、源码或测试（绿场项目），仅使用用户原话和对话上下文作为事实来源，并在开放问题中标注"当前项目无长期文档，后续变更创建后应引用"。

不得静默补全用户没有表达、项目事实也无法证明的业务动机、范围、非范围或影响面。缺失的关键事实必须写入开放问题，并使用苏格拉底追问询问用户。

`lightweight` 类型变更可放宽标准：仅需用户原话和对话上下文作为事实来源。其余类型必须检查全部四项事实来源。

```text
IF proposal 字段需要补充事实才能成立:
  先检查用户原话、当前对话、主 spec、架构文档、ADR、源码、测试或项目文档是否已有依据
ELSE IF 没有事实依据:
  将缺失事实写入开放问题
  使用苏格拉底追问询问用户并暂停
ELSE:
  基于已确认事实填写 proposal
```

## 字段生成规则

- 背景与原因：只写用户明确表达的问题、目标、痛点或已确认背景；缺失时追问。
- 范围 / 包含：只写用户明确要求做的内容，或项目事实能证明属于本次变更的内容。
- 范围 / 不包含：只写用户明确排除的内容；不得为了完整性替用户排除功能。
- 变更类型：AI 可以基于变更性质判断，但必须说明理由。
- 是否需要规约：AI 可以基于是否改变可观察行为判断，但必须引用依据。
- 是否需要技术方案：AI 可以基于影响面和风险判断，但必须引用依据。
- 目标主规约：只能基于用户明确能力、已有 spec 名称或可验证项目事实；不确定时写入开放问题。若"是否需要规约"为 no，填写"不适用"。
  1. 扫描 `specflow/specs/` 下所有主 spec，逐个比对本次变更的需求描述与各主 spec 的能力范围。
  2. 推荐匹配结果：
     - 找到单一匹配 → 写明推荐理由（如"本次变更涉及订单生命周期，与 order.md 的能力范围一致"）
     - 找到多个可能匹配 → 列出候选，各附匹配理由，由用户选择
     - 无匹配 → 建议新建 capability，给出命名建议（2-3 个英文词描述核心行为，非功能名）
  3. 用户确认后写入 proposal 的目标主规约字段。
- 影响面：只写已知或可从项目结构确认的影响；不确定时标注未知或写入开放问题。推断步骤：
  1. 检查 specflow/specs/ 中相关 capability 是否受影响
  2. 检查 specflow/architecture.md 和 adr/ 中相关决策是否受影响
  3. 搜索项目源码中相关模块、API、配置或数据结构
  4. 以上未覆盖的标注为未知
- 开放问题：收纳无法从事实来源确认、且会影响范围、规约、设计或执行的关键问题。
- Roadmap 来源：只记录用户选择的 roadmap F/T 条目；不从 roadmap 静默扩展范围。

## 苏格拉底追问

追问目标是澄清变更边界，而不是替用户决定业务方向。

```text
IF 缺失事实影响范围、是否需要规约、是否需要技术方案、目标主规约或影响面:
  先指出当前无法确定的事实
  再说明该事实为什么影响后续阶段
  最后提出 1-3 个聚焦问题
ELSE:
  不追问，继续整理已确认意图
```

问题应遵守：

- 不诱导用户接受 AI 的默认方案。
- 不一次性抛出大量选项。
- 优先询问会改变范围、是否需要规约或是否需要技术方案的事实。
- 如果用户回答仍然模糊，继续追问边界条件或反例。

示例：

```text
当前需求只说明“增加导出能力”，但没有说明导出对象、格式和权限边界。
这些信息会影响范围、是否需要规约、目标主规约和后续 Spec Delta。

1. 本次导出能力面向普通用户、管理员，还是两者都包含？
2. 导出格式是否已经确定？
3. 是否有明确不做的范围，例如异步导出或批量导出？
```

连续 2 轮追问后用户仍无法给出确定性回答：将该问题记录为阻断性开放问题，告知用户此问题会阻塞后续阶段，待用户主动确认后继续。

## 输出

- `specflow/changes/<change-id>/proposal.md`（使用 `{SKILL_DIR}/assets/proposal.md` 模板创建）

## 变更类型

变更类型直接对应 Tasks 阶段的任务分组，无需额外映射：

- `functional`：改变用户或系统可观察行为。覆盖新功能、功能修改、功能移除、Bug 修复。
- `nonfunctional`：不改变可观察行为，但改变系统质量属性或长期结构。覆盖性能优化、安全加固、重构、技术债清理、可观测性、可靠性提升、系统架构调整（边界、模块职责、数据流、安全模型）、初始项目约定（目录、术语、质量门禁）。
- `infrastructure`：改变构建、运行或部署环境。覆盖依赖升级、框架升级、数据库迁移、CI 调整、容器化、配置变更。
- `lightweight`：不涉及代码逻辑或基础设施变化。覆盖文档、测试补充、合规调整、License、文案、样式、低风险等价替换。

允许组合类型，但只在确实跨类别时使用。

## 是否需要规约判断

```text
IF 变更改变用户或系统可观察行为:
  是否需要规约 = yes
ELSE:
  是否需要规约 = no，并说明理由
```

可观察行为包括 UI 行为、API 响应、CLI 参数、错误语义、状态变化、数据保留策略、同步或一致性语义。

## 是否需要技术方案判断

```text
IF 变更涉及跨模块影响、API/CLI/数据模型/状态流变化、架构边界、安全、部署、外部依赖、多方案取舍、高返工风险或长期约束:
  是否需要技术方案 = yes
ELSE:
  是否需要技术方案 = no，并说明理由
```

## 阻断

"不清楚"的判定标准：无法从已定义的四个事实来源（用户原话、对话上下文、长期项目事实、可验证源码事实）中唯一确定该字段的取值。

阻断性开放问题 = 其答案会改变范围、是否需要规约、是否需要技术方案、目标主规约或影响面。其余为记录性开放问题：可写入 proposal 带入后续阶段，不阻断。

```text
IF 范围 / 不包含 不清楚:
  使用苏格拉底追问询问用户并暂停
ELSE IF 是否需要规约 或 是否需要技术方案 判断缺少理由:
  补齐理由
ELSE IF 存在阻断性开放问题:
  使用苏格拉底追问询问用户并暂停
ELSE IF proposal 包含没有事实来源的新业务动机、范围、非范围或影响面:
  暂停，移入开放问题并使用苏格拉底追问询问用户
```

## 完成条件

- `proposal.md` 存在，且 Roadmap 来源已记录；没有来源时明确写“无”。
- 范围 / 不包含 明确。
- 变更类型、是否需要规约、是否需要技术方案均有结论和理由。
- 无阻断性开放问题。
- 无未标明事实来源的新业务动机、范围、非范围或影响面。
