# Design Stage

本文件定义技术实现设计阶段。Design 说明如何在既有 System Architecture / ADR 约束下实现 proposal 和 spec-delta，不重新定义用户需求，也不临场决定系统架构图或长期 ADR。

路径锚定：本文件中的 `specflow/`、源码、测试、配置和部署路径均相对于 `{PROJECT_ROOT}/`；不得在 `{SKILL_DIR}/` 下查找 proposal、spec-delta、design 或长期项目文档。

## 目标

- 从已确认的 proposal 和 spec-delta 转入技术实现设计，回答"怎么做"。
- 根据变更类型选择对应族插槽，系统性地覆盖该族的所有设计维度。
- 发现本次实现是否需要新增或调整系统边界图、系统架构图或 ADR。
- 所有 UML 使用 Mermaid 表达。
- 避免只画图而忽略数据设计和异常处理——这两者是线上故障的重灾区。

## 输入

- `specflow/changes/<change-id>/proposal.md` — **必读**，提取变更类型（functional / nonfunctional / infrastructure / lightweight）
- `specflow/changes/<change-id>/spec-delta.md` — 功能族必需
- 相关主 spec、`specflow/architecture.md`、`specflow/adr/`、源码、测试、配置和部署事实

## 输出

- `specflow/changes/<change-id>/design.md`

## 族选择规则

```text
读取 proposal.md 的变更类型字段：
  functional    → 功能族插槽（§1-§8）
  nonfunctional → 优化族插槽（§1-§5）
  infrastructure → 迁移族插槽（§1-§6）
  lightweight   → 不进入 Design 阶段（轻量族无需技术方案）
```

## 设计维度与质量标准

### §0. 变更上下文（公共前置，所有族共享）

**目的**：在设计开始前，明确变更的约束和影响范围，为后续所有维度提供上下文。

**必须覆盖**：

- **变更类型**：从 proposal 继承（functional / nonfunctional / infrastructure）
- **上游约束**：从 architecture.md / ADR 中提取本变更必须遵守的约束
- **影响面摘要**：改动涉及哪些模块、接口、数据、配置

**质量检查**：接手者只看这一节，就能理解本次设计的前提条件和约束边界。

### 功能族插槽（§1-§8）

功能族的核心不确定性是"怎么实现"——从需求到代码的完整路径。

#### §1. 架构与模块定位

**目的**：明确功能在系统整体中的位置，界定模块边界和分层职责。

**必须覆盖**：

- **模块划分**：该功能属于哪个已有模块，或需要新建什么模块。定义模块间的静态依赖关系。
- **分层职责**：在分层架构（Controller、Service、DAO、第三方适配层）中，清晰界定各层负责的具体逻辑，避免跨层耦合。
- **核心设计模式**（如适用）：说明会用到的主要设计模式（如策略、工厂、观察者等），并解释其在该功能中的具体作用。简单 CRUD 功能可写"本功能无特殊设计模式"。

**常见遗漏点**：

- 只描述了功能本身，没说明它和已有模块的关系
- 分层职责模糊，把所有逻辑都堆在 Service 层

**质量检查**：新人只看这一节，能画出该功能在系统中的位置和调用链路。

#### §2. 核心流程 UML

**目的**：用图表达核心流程，作为后续实现的统一认知基准。

**UML 选择指南**：

| UML 图 | 作用 | 选择条件 |
|--------|------|---------|
| 活动图/流程图（flowchart） | 表达业务决策、分支、并行、循环 | 存在复杂判断逻辑（如订单状态机流转、多条件分支） |
| 时序图（sequenceDiagram） | 表达对象间交互时序和调用顺序 | ≥2 个组件协作（如 Controller→Service→DAO→缓存→MQ） |
| 状态图（stateDiagram-v2） | 表达核心实体的生命周期状态变迁 | 实体有状态变迁且触发条件有业务规则约束 |
| 用例图（flowchart 表达） | 辅助确认设计覆盖所有需求场景 | 需求规约中的角色或场景复杂 |
| ER 图（erDiagram） | 表达实体关系 | 数据模型复杂，多表关联 |
| 文字 + 表格 | — | 流程简单，画图反而增加噪音 |

选择时：先判断是否需要 UML，再选最匹配的类型。不要为了画图而画图。一个功能可能同时需要活动图（表达业务分支）+ 时序图（表达调用链），按需组合。

**必须覆盖**：

- 如果核心流程涉及 ≥2 个组件协作或有分支判断，必须用至少一张图表达核心流程；流程简单时文字 + 表格即可
- 时序图必须真实反映调用层次，**禁止把多个层级压缩在一条生命线中**——Controller 和 Service 和 DAO 必须各自独立生命线
- 状态图必须标注所有合法变迁和触发条件

**常见遗漏点**：

- 只画了流程图但没有时序图，导致组件间交互不清
- 时序图把 Controller/Service/DAO 合并成一条"后端"生命线——这等于没画
- 状态图缺少非法变迁的标注（如「已完成状态不允许退回草稿」）

**质量检查**：时序图中的每条生命线对应一个真实的类/模块/服务。

#### §3. 数据设计

**目的**：确保数据层设计完整，避免上线后发现数据模型缺陷、缓存击穿、消息丢失。

**⚠️ 本节是线上故障的重灾区，不得跳过或弱化为一句「沿用现有数据模型」或「本功能无数据变更」。**

**3a. 数据模型** — 以实体为核心，与存储形式无关：

数据模型描述的是「有哪些数据单元、各自有什么属性、如何关联、如何访问、如何演进」。这些问题的答案不依赖于存储形式（关系型数据库、JSON 文件、文档数据库、KV 存储）。不得以存储形式特殊为由跳过任何一项。

- **实体清单**：有哪些数据单元？每个实体当前的存储形式（表名 / 文件路径 / 集合名）？
- **属性定义**：每个实体包含哪些属性？类型、是否必填、默认值、约束？
- **实体关系**：实体之间如何关联（1:1、1:N、N:M）？通过什么机制实现（外键 / 嵌入 / 引用路径）？
- **访问路径**：常见查询/读取模式是什么？存储层如何高效支持（数据库索引 / 目录结构 / 文件命名规则）？
- **并发控制**：哪些实体存在并发写冲突？用什么策略（悲观锁 / 乐观锁 / 文件锁）？冲突如何恢复？
- **数据迁移**：属性新增/删除/类型变更时，已有数据如何兼容？

> 以上 6 个子项均需填写。如果某子项确实不适用，写"不适用"并说明理由，不能静默跳过。

> ER 图（Mermaid erDiagram）是存储无关的，适用于所有存储形式。

**3b. 缓存设计**（本功能主动读写缓存时适用）：

如本功能不主动读写缓存（仅经过已有缓存中间件不算），写「本功能不涉及缓存」并说明理由。不要静默跳过。

- 哪些数据需要缓存？为什么（读多写少？热点？计算昂贵？）
- 缓存 Key 格式（如 `order:detail:{orderId}`）
- 过期策略（TTL、LRU、主动失效）和过期时间
- 更新/失效时机（写后失效还是写后更新？）
- 缓存穿透/击穿/雪崩的防范（如布隆过滤器、互斥锁、随机 TTL）

**3c. 消息/事件设计**（本功能主动生产或消费异步消息时适用）：

如本功能不涉及异步消息，写「本功能不涉及异步消息」并说明理由。不要静默跳过。

- Topic / Queue 名称
- 消息体结构（JSON Schema 或字段说明）
- 生产者逻辑（何时发送、发送失败处理）
- 消费者逻辑（消费成功/失败的后续动作）
- 重试、死信队列、幂等消费机制

**常见遗漏点**：

- 只列了实体和属性，没有定义实体关系
- 只写了缓存 Key，没有过期策略和失效时机
- 涉及异步但没有消息体结构定义
- 没有考虑缓存穿透/击穿/雪崩
- 以存储形式特殊（JSON 文件、文档数据库）为由跳过整个数据设计章节

**质量检查**：接手者只看这一节就能完成数据层的创建（无论存储形式是表、文件还是集合），并正确配置缓存和消息通道。

#### §4. 接口设计

**目的**：让上下游能基于接口定义并行开发，减少集成阶段的沟通成本。

**4a. 对外接口**（REST API、RPC 或消息通道）：

- 协议类型（HTTP/RPC/WebSocket）
- 路径和方法（如 `POST /api/v1/order/calculate`）
- 请求结构（Header、Path Param、Query Param、Body）
- 响应结构（成功和失败两种格式）
- **错误码清单**：覆盖该接口所有可预见的失败模式。至少包括：参数校验失败、资源不存在（如适用）、权限不足（如适用）、服务不可用（如适用）、未知错误。错误码必须含 HTTP 状态码、含义、触发条件和调用方处理建议。
- 鉴权方式（Token、签名、OAuth）
- **接口版本兼容策略**（如适用）：新增字段时如何兼容旧版本

**4b. 模块间交互契约**：

- 各层之间或模块间的交互方向和数据格式
- 参数类型和含义
- 返回值类型和含义
- 异常/错误传播策略（抛出什么异常、调用方如何处理）

**4c. 第三方接口**（如调用外部服务）：

如不涉及外部服务调用，写「本功能不涉及第三方接口」。

- 依赖的服务名称和接口
- 超时时间（连接超时、读取超时）
- 重试策略（重试次数、退避算法、可重试的错误类型）
- 降级策略（调用失败时的兜底行为）
- 熔断策略（熔断阈值、半开探测、恢复条件）

**常见遗漏点**：

- 只写了成功响应结构，没有失败响应和错误码清单——这会导致上下游联调时反复沟通
- 模块间交互只写了方法名没有异常约定
- 第三方接口没有超时和重试策略，默认值上线后才发现不合理

**质量检查**：前端/下游只看这一节就能写出完整的调用代码和错误处理逻辑。

#### §5. 关键业务规则与实现逻辑

**目的**：让有实现歧义的规则在编码前就有明确的逻辑表达。

**"关键规则"的判断很简单**：spec-delta 需求使用了带触发条件或条件分支的 EARS 句式——即 `WHEN <trigger> THE system SHALL`、`WHILE <state> THE system SHALL`、`WHERE <feature> THE system SHALL`、`IF <condition>, THEN THE system SHALL`——这些就是关键规则信号，必须纳入 §5 展开。Ubiquitous 句式（`THE system SHALL`，无条件）一般不是关键规则，除非 `<response>` 本身描述了一段计算或判定逻辑。也兼容识别中文残留表述如「根据…计算」「按照…规则」「当…时则」「满足…条件时」「如果…那么…」。

如果不确定要不要写——就写。宁可多写一条伪代码，也不要让两个开发者对同一条规则做出两种实现。

**必须覆盖**：

- 规则描述：自然语言 + 至少 2 个具体示例（含正常值和边界值）
- 伪代码或逻辑表达式
- 方案对比（如适用——只有当确实存在多种实现路径时才填写）
- 推荐方案与理由

**如果确实无关键规则**：写「本功能无关键业务规则」并引用 spec-delta 原文说明判断依据。

**常见遗漏点**：

- 规则描述只有一句话，缺少具体示例——比如「根据用户等级给予折扣」没写各等级的具体折扣
- 以为规则简单就不用写——一条 if/else 写出来也比两个人各自理解强
- 以「本功能无关键算法」跳过，但未引用 spec-delta 原文佐证

**质量检查**：另一个开发者只看这一节，就能写出正确的实现，且边界行为一致。

#### §6. 异常处理与边界条件

**目的**：提前想清楚「出错了怎么办」，避免线上故障时临时救火。

**必须覆盖**：

- **异常场景列表**：至少覆盖数据非法（参数校验失败、数据格式错误）、依赖失败（数据库宕机、缓存不可用、第三方超时）、并发冲突（乐观锁失败、死锁）、业务异常（库存不足、余额不足、状态不允许）。每条异常标注对应错误码（见 §4a 错误码清单）。
- **处理策略**：每种异常是重试、降级、拒绝还是补偿？重试需要幂等支持吗？
- 日志与告警：哪些异常需要 ERROR 级别日志？哪些需要触发告警？
- **边界条件**：分页极限值、空列表、null 参数、重复提交、超大数据量、时间为零值/负值
- **幂等性设计**：哪些操作需要幂等？用什么机制实现（唯一键、Token、状态机）？不需要幂等的操作（如只读操作）写"不需要"并说明理由。
- 并发冲突处理：乐观锁 vs 悲观锁，选型和理由

**常见遗漏点**：

- 只列了「库存不足」这种业务异常，没考虑「数据库连接超时」这种基础设施异常
- 提了幂等但没有说具体实现机制
- 边界条件只考虑了正常范围，没考虑极端值（0、负数、超长字符串、极限并发）
- 异常场景与 §4 错误码清单未建立对应关系

**质量检查**：运维只看这一节就知道哪些异常需要告警、哪些可以自动恢复。

#### §7. 非功能性设计

**目的**：确保功能上线后不只是「能用」，而是「好用、安全、可维护」。

**必须覆盖**：

- **性能**：量级预估（10/100/1000/10000+ QPS），关键路径的响应时间目标（P50/P99），是否需异步化或批量处理
- **安全**：权限校验点（哪些接口需要什么权限）、敏感数据脱敏（日志/响应中哪些字段需要脱敏）、防注入（SQL 注入、XSS）、接口限流
- **监控与日志**：关键步骤的日志输出点（如"生成订单开始"、"生成订单成功"）、业务指标埋点（如"订单创建成功"计数器）、链路追踪 ID 的传递
- **配置**：提取到配置中心的开关、阈值、参数（如"是否开启新算法"、"优惠券最大叠加数"）
- **限流/降级/批处理参数**（如适用）：不适用时写"本功能不涉及"。

**常见遗漏点**：

- 性能预估写「无特殊要求」但实际可能有热点
- 安全只写「需要登录」但没有具体权限校验点
- 监控只写了「记录日志」但没有具体埋点位置和指标名称

**质量检查**：SRE 只看这一节就能配置告警规则和监控面板。接手者能理解本次设计的核心决策和取舍。

#### §8. 测试与可验证性设计

**目的**：为测试阶段提供明确的覆盖目标，不等到测试时才发现设计遗漏。

**必须覆盖**：

- **正常场景**：覆盖 spec-delta 中的所有 Gherkin Scenario
- **异常场景**：至少覆盖 §6 中的 3 个关键异常
- **边界场景**：至少覆盖空值、极限值、并发
- 测试数据构造方式（如何准备测试数据、如何清理）
- Mock 点（哪些外部依赖需要 Mock，精确到具体类和方法，如"Mock UserDAO.findById"）

**常见遗漏点**：

- 只写了正常场景，没写异常和边界
- Mock 点模糊——「Mock 数据库」太粗糙，应该精确到「Mock UserDAO.findById」
- "spec-delta 来源"列未标注或标注不准确

**质量检查**：测试工程师只看这一节就能写出完整的测试用例清单。

### 优化族插槽（§1-§5）

优化族的核心不确定性不是"怎么落地"，而是"优化后真的更好吗？会不会改坏？"。

#### §1. 优化目标与基线

**目的**：明确优化的具体方向、当前基线和目标指标——没有基线就没有"优化了"的判据。

**必须覆盖**：

- **优化目标**：从 proposal 继承（性能 / 安全 / 重构 / 技术债 / 可观测性 / 可靠性），明确优化的具体方向和预期效果
- **当前基线指标**：每个优化方向的当前指标值、测量方法、测量条件
- **目标指标**：每个优化方向的目标值和达标判据

**质量检查**：接手者能用 §1b 的测量方法复现基线数据，能用 §1c 的达标判据判定优化是否成功。

#### §2. 优化方案设计

**目的**：对比多种优化方案，给出推荐方案和选择理由——优化族的核心决策是"为什么选这个方案"。

**必须覆盖**：

- **方案对比**：每个优化方向列出候选方案，对比预期提升、风险、复杂度、副作用
- **推荐方案与理由**：为什么选这个方案而不是那个
- **影响面分析**：改动涉及哪些模块、接口、数据结构、配置

**质量检查**：另一个工程师能理解为什么推荐这个方案，并能评估替代方案。

#### §3. 实现设计

**目的**：把推荐方案细化为可落地的实现结构。

**必须覆盖**：

- **核心改动**：关键代码结构变更、算法替换、架构调整的具体实现方式
- **数据变更**（如适用）：缓存策略调整、索引变更、数据结构重构。不适用时写"本优化不涉及数据变更"。

**质量检查**：开发者只看这一节就能开始编码实现。

#### §4. 风险与回滚

**目的**：提前识别优化引入的新风险，并准备回退方案——"优化本身有问题怎么办"。

**必须覆盖**：

- **优化引入的新风险**：每个风险的触发条件、影响范围、防范措施
- **行为回归风险**：哪些已有行为可能受影响
- **回滚方案**：如果优化效果不达预期或引入问题，如何回退

**质量检查**：运维知道优化失败时如何回退，知道哪些已有行为需要重点验证。

#### §5. 验证策略

**目的**：为 Tasks 阶段的"指标达标 + 行为回归"验证提供设计依据。

**必须覆盖**：

- **行为回归测试**：运行已有测试，确认无行为变更。明确哪些测试需要运行。
- **指标对比测试**：对比 §1b 基线值和优化后值，判定是否达到 §1c 目标
- **边界与极端场景**：并发、大数据量、资源耗尽等优化可能引入的极端场景

**质量检查**：测试工程师只看这一节就能设计出指标对比测试和行为回归测试方案。

### 迁移族插槽（§1-§6）

迁移族的核心不确定性不是"怎么改代码"，而是"迁移过程中出问题怎么办？"。

#### §1. 迁移目标与范围

**目的**：明确迁移的目标、范围和约束——迁移的设计完全由约束驱动。

**必须覆盖**：

- **迁移目标**：从 proposal 继承（升级 / 迁移 / CI / 容器化 / 配置变更）
- **迁移范围**：每个组件的当前状态、目标状态、迁移类型
- **迁移约束**：停机时间限制、兼容性要求、数据量、时间窗口

**质量检查**：接手者能判断迁移方案是否在约束范围内。

#### §2. 迁移流程设计

**目的**：把迁移定义为一个有阶段、可回滚的过程——迁移本质上是过程，不是功能。

**必须覆盖**：

- **迁移阶段**：每个阶段的操作、耗时估计、是否可回滚、验证方法
- **迁移流程图**：用 Mermaid flowchart 表达迁移全过程
- **切换策略**：蓝绿 / 滚动 / 影子 / 停机切换

**质量检查**：运维只看这一节就能执行迁移操作并知道当前在哪个阶段。

#### §3. 数据迁移设计

**目的**：确保数据迁移的转换规则明确、一致性可校验、失败可回退。

**必须覆盖**：

- **数据转换规则**：源结构到目标结构的转换逻辑、数据量估计
- **数据一致性校验**：校验方法、校验工具、校验时机、不一致时的处理
- **数据回滚方案**：迁移失败后数据如何回退

**质量检查**：DBA / 数据工程师只看这一节就能执行数据迁移并验证一致性。

#### §4. 兼容性设计

**目的**：确保迁移期间新旧版本能共存——迁移进行到一半时系统仍能正常工作。

**必须覆盖**：

- **新旧版本并存**：迁移期间新旧版本如何共存
- **接口兼容性**（如适用）：迁移期间接口是否保持兼容
- **配置兼容性**（如适用）：新旧配置格式如何兼容

**质量检查**：接手者能判断迁移期间系统是否能正常运行。

#### §5. 异常处理与回滚

**目的**：提前想清楚迁移过程中的各种失败场景和回退方案——迁移的关键难题是"部分完成"状态。

**必须覆盖**：

- **迁移失败场景**：每个场景的触发条件、处理策略、回滚步骤
- **回滚计划**：回滚触发条件、回滚步骤、回滚时间估计、回滚验证
- **部分迁移失败处理**：某个阶段失败但前面阶段已完成时如何处理

**质量检查**：运维知道迁移失败时如何回退，知道"迁移到一半"时怎么处理。

#### §6. 验证策略

**目的**：为 Tasks 阶段的"迁移完成 + 行为回归 + 基线对比"验证提供设计依据。

**必须覆盖**：

- **迁移完成验证**：数据一致性、服务连通性、功能可用性的验证方法
- **行为回归测试**：迁移后行为不变的验证方法
- **性能基线对比**：迁移后性能不退化的验证方法

**质量检查**：测试工程师只看这一节就能设计出迁移验证方案。

### §9. System Architecture / ADR 影响（公共收尾，所有族共享）

**目的**：汇总本次设计对系统架构图和 ADR 的影响，确保不遗漏长期文档变更。

**架构影响判定**：

```text
IF 技术内容只是本次实现的内部组织方式:
  写入对应设计维度，不在 §9 中记录架构变更
ELSE IF 技术内容改变系统外部关系、外部系统、用户/角色、信任边界、主要模块/服务/容器、数据存储、关键集成或部署边界:
  在 §9 中记录架构变更
```

内部 service、class、function、helper、目录名或文件名本身不自动构成架构影响。只有当它改变系统边界图或系统架构图时，才升级为架构影响。

**ADR 影响判定**：

```text
IF 决策属于技术栈选型、核心架构约束、可观测性、数据所有权与存储策略、认证/安全方案、关键集成方案或 AI 协作禁止项:
  在 §9 中记录 ADR 候选描述
ELSE:
  只记录在 design.md
```

Design 不创建或修改 `specflow/architecture.md` 或 ADR。系统架构和 ADR 的首次创建、后续更新、替代或废弃均在 System Architecture / ADR 阶段完成。AI 的推荐方案、推断、草案或沉默通过均不等同于用户确认。

**分族架构影响检查清单**：

<!-- 通用判定已覆盖所有族。以下清单帮助不同族的设计者检查典型触发点，避免遗漏。 -->

**功能族典型触发点：**
- 新增或修改了对外接口协议（REST/RPC/WebSocket）→ 系统边界图
- 新增了用户角色或信任边界 → 系统边界图
- 新增了外部系统对接 → 系统边界图 + 系统架构图
- 新增了独立部署的模块/服务/容器 → 系统架构图
- 选定了新的技术栈或框架 → ADR 候选
- 涉及认证/安全方案变更 → ADR 候选

**优化族典型触发点：**
- 替换或引入新的中间件（缓存/队列/搜索引擎）→ 系统架构图 + ADR 候选
- 调整了模块边界或分层职责 → 系统架构图
- 改变了数据存储策略（如从数据库改为文件存储）→ 系统架构图 + ADR 候选
- 引入了新的可观测性方案（监控/链路追踪/日志规范）→ ADR 候选
- 改变了数据所有权（哪个模块拥有哪份数据的读写权）→ ADR 候选

**迁移族典型触发点：**
- 替换了数据库/框架/运行时 → 系统架构图 + ADR 候选
- 改变了部署架构（如从虚拟机到容器）→ 系统架构图 + ADR 候选
- 改变了 CI/CD 基础设施 → ADR 候选
- 改变了配置管理方式 → ADR 候选
- 迁移过程中系统边界图暂时性变化（新旧并存期间）→ 系统边界图

**必须输出**：

- 需要进入 System Architecture / ADR 处理的系统边界图或系统架构图变更（无变更写「无」）
- 需要进入 System Architecture / ADR 处理的 ADR 候选（无 ADR 需求写「无」）
- 用户自述选项

## 讨论循环

```text
  读取当前 change 目录中的 proposal、spec-delta（功能族），以及主 spec、architecture、ADR、源码和配置事实

  填写 §0 变更上下文：从 proposal 提取变更类型，从 architecture/ADR 提取约束，梳理影响面

  根据 proposal 的变更类型选择族插槽：
    functional    → 功能族（§1-§8）
    nonfunctional → 优化族（§1-§5）
    infrastructure → 迁移族（§1-§6）

  遍历上游规约，记录待覆盖清单：
    功能族：遍历 spec-delta 所有场景和关键约束
    优化族：遍历 proposal 中的优化目标
    迁移族：遍历 proposal 中的迁移范围

  识别是否需要新增或调整系统边界图、系统架构图或 ADR

  按选中族插槽的维度顺序逐项展开技术实现方案：
    FOR EACH 维度：

      进入下一维度

  检查所有维度是否完整：
    功能族：特别确认"数据设计"和"异常处理与边界条件"未被弱化
    优化族：特别确认"基线指标"和"回滚方案"未被弱化
    迁移族：特别确认"迁移流程"和"数据一致性校验"未被弱化

  验证全覆盖：
    功能族：
      FOR EACH spec-delta 场景:
        IF §8 存在对应测试用例且标注了 spec-delta 来源: 标记已覆盖
        ELSE: 暂停，补齐缺失的测试用例
      FOR EACH spec-delta 关键约束:

    优化族：
      FOR EACH §1 优化目标:

        ELSE: 暂停，补齐缺失的验证策略
    迁移族：
      FOR EACH §2 迁移阶段:
        IF §6 存在对应验证方法: 标记已覆盖
        ELSE: 暂停，补齐缺失的验证方法

  汇总架构与 ADR 影响：
    FOR EACH 当前族所有维度的讨论结果:
      IF 描述了需要修改系统边界图、系统架构图，或涉及 ADR 适用范围中的长期决策:
        提取对应描述，对照 §9 分族架构影响检查清单，写入 §9

  将完整 design.md（§0 + 族插槽 + §9 + 开放问题）写盘

  IF §9 无架构或 ADR 变更（全部为"无"）:
    IF 无开放问题:
      告知用户 "Design 阶段已完成，可进入 Tasks 阶段：/specflow-workflow tasks"
  ELSE：
    告知用户 "Design 阶段已完成。存在架构/ADR 变更需求，请先执行 /specflow-workflow arch"
```

## 开放问题

设计过程中产生的、因依赖未满足或事实缺失而无法当场解决的缺口，写入 design.md 末尾"开放问题"节。

### 来源

| 族 | 维度 | 可能产生开放问题的场景 |
|---|---|---|
| 功能族 | §1 架构与模块定位 | 跨模块调用方式依赖尚未创建的 ADR；模块边界与已有模块冲突且冲突方未确认 |
| 功能族 | §2 核心流程 UML | 流程分支依赖未敲定的业务规则 |
| 功能族 | §3 数据设计 | 存储形式依赖未选型的基础设施；第三方数据源 schema 未确认 |
| 功能族 | §4 接口设计 | 外部服务选型未定，超时、重试、降级策略无法确定；鉴权方案依赖未确认的安全 ADR |
| 功能族 | §5 关键业务规则 | spec-delta 或 proposal 中的规则表述有歧义，无法唯一确定实现 |
| 功能族 | §6 异常处理与边界条件 | 降级策略依赖未确认的基础设施容错能力；幂等机制依赖未确认的外部系统行为 |
| 功能族 | §7 非功能性设计 | 用户未给出性能指标或安全要求；SLI/SLO 依赖未确认的监控基础设施 |
| 功能族 | §8 测试与可验证性设计 | Mock 点依赖未确认的依赖项行为 |
| 优化族 | §1 优化目标与基线 | 基线数据无法测量或测量工具缺失 |
| 优化族 | §2 优化方案设计 | 方案选型依赖未确认的外部决策 |
| 优化族 | §4 风险与回滚 | 回滚方案依赖未确认的运维能力 |
| 迁移族 | §1 迁移目标与范围 | 迁移约束（停机时间、兼容性）未确认 |
| 迁移族 | §2 迁移流程设计 | 迁移阶段依赖未确认的外部协调 |
| 迁移族 | §3 数据迁移设计 | 数据一致性校验工具未选型 |
| 迁移族 | §4 兼容性设计 | 新旧并存方案依赖未确认的基础设施能力 |
| 迁移族 | §5 异常处理与回滚 | 回滚方案依赖未确认的备份策略 |

### 处理

```text
FOR EACH 当前族各维度讨论中产生的缺口:
  IF 缺口不影响其他维度的设计推进:
    记录为记录性开放问题 → 写入 design.md "开放问题"节，带入后续阶段
  ELSE IF 缺口导致后续维度无法继续讨论:
    记录为阻断性开放问题 → 询问用户 → 暂停该维度
    2 轮追问后仍无法确认 → 标记为阻断，写入 design.md 并告知用户
```

### 与 §9 的关系

开放问题中属于"依赖架构或 ADR 确认才能继续设计"的条目，同时反映在 §9 的架构/ADR 变更描述中。§9 是架构影响的汇总，开放问题是 Design 自身缺口的汇总——两者可能重叠但不互相替代。

## 避坑约束

以下规则在讨论循环和输出 design.md 时强制执行：

1. **禁止只画 UML 而忽略数据设计和异常处理**：数据设计和异常处理是线上故障的两大重灾区。如果在讨论循环中数据设计或异常处理的内容明显弱于 UML，暂停并补齐。（功能族适用）
2. **时序图必须真实反映调用层次**：Controller、Service、DAO 必须各自独立生命线。禁止把多个层级合并到一条生命线中。（功能族适用）
3. **接口设计必须包含完整错误码清单**：设计 §4 的对外接口必须列出覆盖所有可预见失败模式的错误码及其含义。错误码必须含 HTTP 状态码、含义、触发条件和调用方处理建议。缺少错误码清单的设计视为不完整。（功能族适用）
4. **数据设计不得因存储形式特殊而跳过或弱化**：设计 §3 必须覆盖实体、属性、关系、访问路径、并发控制和数据迁移，无论存储形式是表、文件还是集合。如果只覆盖了数据模型而跳过了缓存和消息/事件（当适用时），暂停并补齐。（功能族适用）
5. **讨论循环不可跳过任何维度**：即使某个维度看起来简单或无内容，也必须明确写「本功能不涉及 X」并说明理由，不能静默跳过。（所有族适用）
6. **§5 不得以"无关键业务规则"为由跳过而未引用 spec-delta 原文佐证**：如果功能族 §5 为空，必须确认 spec-delta 中不存在需要「算」或「判」的表述。（功能族适用）
7. **§9 架构影响检查不得因族类型而跳过**：优化族和迁移族的设计者容易只关注优化指标或迁移流程，忽略对系统架构图和 ADR 的影响。必须对照 §9 中的分族架构影响检查清单逐项检查，即使全部为"无"也要显式记录。（所有族适用）

## 阻断

```text
IF proposal 的"是否需要技术方案"为 yes 但没有当前 change 目录中的 design.md:
  暂停并在 {PROJECT_ROOT}/specflow/changes/<change-id>/ 创建 design.md
ELSE IF 缺少 architecture.md，且本次变更需要生成代码或确定实现结构:
  告知用户 "项目缺少 architecture.md，请先执行 /specflow-workflow arch"
ELSE IF design.md §9 包含非"无"的 ADR 候选描述但对应 ADR 缺失或未确认:
  告知用户 "对应 ADR 缺失，请先执行 /specflow-workflow arch"
ELSE IF 功能族：design 与 spec-delta 描述的行为冲突:
  暂停并修正冲突
ELSE IF design 引入系统图变更或 ADR 适用范围内的长期决策:
  告知用户 "Design 阶段已完成。存在架构/ADR 变更需求，请先执行 /specflow-workflow arch"
ELSE IF design 包含未决关键问题:
  询问用户并暂停
```

> 维度内容完整性由讨论循环的逐维度自检和完成条件保障，不在此处重复。

## 审查

design.md 写入后，按 SKILL.md「阶段产出物审查」启动独立审查-修复-审查循环（最多三轮），委托审查 agent 检查：

**所有族通用审查项**：
- 族插槽选择与 proposal 变更类型是否一致
- §0 变更上下文是否完整
- §9 架构影响是否与 architecture.md / ADR 一致
- §9 分族架构影响检查清单是否逐项检查（不能因族类型跳过）

**功能族专项审查项**：
- 与 spec-delta 行为一致性
- 功能族维度覆盖完整性（§1-§8：模块分层、核心流程、数据设计、接口设计、关键业务规则、异常处理、非功能需求、测试策略）
- 数据设计是否覆盖实体、属性、关系、访问路径、并发控制、迁移策略
- 接口错误码是否覆盖所有可预见失败模式且含义明确
- §6 异常场景是否与 §4 错误码建立交叉引用

**优化族专项审查项**：
- §1 基线指标是否可测量、§1c 目标指标是否有达标判据
- §2 方案对比是否给出推荐理由
- §4 回滚方案是否可行
- §5 验证策略是否覆盖行为回归和指标对比

**迁移族专项审查项**：
- §2 迁移阶段是否完整且每阶段可回滚
- §3 数据一致性校验方法是否明确
- §4 兼容性方案是否可行
- §5 回滚计划是否完整、部分失败处理是否覆盖
