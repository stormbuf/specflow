# Design Stage

本文件定义功能技术方案阶段。Design 说明如何在既有 System Architecture / ADR 约束下实现 spec-delta，不重新定义用户需求，也不临场决定系统架构图或长期 ADR。

路径锚定：本文件中的 `specflow/`、源码、测试、配置和部署路径均相对于 `{PROJECT_ROOT}/`；不得在 `{SKILL_DIR}/` 下查找 proposal、spec-delta、design 或长期项目文档。

## 目标

- 从已确认的功能规约转入功能技术实现设计，回答"怎么做"。
- 系统性地覆盖 8 个设计维度，形成可落地的技术设计文档。
- 发现本次实现是否需要新增或调整系统边界图、系统架构图或 ADR。
- 所有 UML 使用 Mermaid 表达。
- 避免只画图而忽略数据设计和异常处理——这两者是线上故障的重灾区。

## 输入

- `specflow/changes/<change-id>/proposal.md`
- `specflow/changes/<change-id>/spec-delta.md`，如果"是否需要规约"为 yes
- 相关主 spec、`specflow/architecture.md`、`specflow/adr/`、源码、测试、配置和部署事实

## 输出

- `specflow/changes/<change-id>/design.md`

## 设计维度与质量标准

讨论循环中，按以下 8 个维度逐项展开。每完成一个维度，对照该维度的质量标准和常见遗漏点做自检，检查通过才进入下一维度。

### 1. 架构与模块定位

**目的**：明确功能在系统整体中的位置，界定模块边界和分层职责。

**必须覆盖**：
- **模块划分**：该功能属于哪个已有模块，或需要新建什么模块。定义模块间的依赖关系。
- **分层职责**：在分层架构（Controller、Service、DAO、第三方适配层）中，清晰界定各层负责的具体逻辑，避免跨层耦合。
- **核心设计模式**：说明会用到的主要设计模式（如策略、工厂、观察者等），并解释其在该功能中的具体作用。

**常见遗漏点**：
- 只描述了功能本身，没说明它和已有模块的关系
- 分层职责模糊，把所有逻辑都堆在 Service 层
- 提了设计模式但没有解释「为什么在这个场景用」

**质量检查**：新人只看这一节，能画出该功能在系统中的位置和调用链路。

### 2. 核心流程 UML

**目的**：用图表达核心流程，作为后续实现的统一认知基准。

**UML 选择指南**：

| UML 图 | 作用 | 选择条件 |
|--------|------|----------|
| 活动图/流程图（flowchart） | 表达业务决策、分支、并行、循环 | 存在复杂判断逻辑（如订单状态机流转、多条件分支） |
| 时序图（sequenceDiagram） | 表达对象间交互时序和调用顺序 | 多组件协作（如 Controller→Service→DAO→缓存→MQ） |
| 状态图（stateDiagram-v2） | 表达核心实体的生命周期状态变迁 | 实体状态超过 3 种且变迁有明确规则（如审批、工单） |
| 用例图（flowchart 表达） | 辅助确认设计覆盖所有需求场景 | 需求规约中的角色或场景复杂 |
| ER 图（erDiagram） | 表达实体关系 | 数据模型复杂，多表关联 |
| 文字 + 表格 | — | 流程简单，画图反而增加噪音 |

选择时：先判断是否需要 UML，再选最匹配的类型。不要为了画图而画图。一个功能可能同时需要活动图（表达业务分支）+ 时序图（表达调用链），按需组合。

**必须覆盖**：
- 至少用一张图表达核心流程
- 时序图必须真实反映调用层次，**禁止把多个层级压缩在一条生命线中**——Controller 和 Service 和 DAO 必须各自独立生命线
- 状态图必须标注所有合法变迁和触发条件

**常见遗漏点**：
- 只画了流程图但没有时序图，导致组件间交互不清
- 时序图把 Controller/Service/DAO 合并成一条"后端"生命线——这等于没画
- 状态图缺少非法变迁的标注（如「已完成状态不允许退回草稿」）

**质量检查**：时序图中的每条生命线对应一个真实的类/模块/服务。

### 3. 数据设计（核心内容，最易遗漏）

**目的**：确保数据层设计完整，避免上线后发现数据模型缺陷、缓存击穿、消息丢失。

**⚠️ 本节是线上故障的重灾区，不得跳过或弱化为一句「沿用现有数据模型」或「本功能无数据变更」。**

**3a. 数据模型** — 以实体为核心，与存储形式无关：

数据模型描述的是「有哪些数据单元、各自有什么属性、如何关联、如何访问、如何演进」。这些问题的答案不依赖于存储形式（关系型数据库、JSON 文件、文档数据库、KV 存储）。不得以存储形式特殊为由跳过任何一项。

- **实体清单**：有哪些数据单元？每个实体当前的存储形式（表名 / 文件路径 / 集合名）？
- **属性定义**：每个实体包含哪些属性？类型、是否必填、默认值、约束？
- **实体关系**：实体之间如何关联（1:1、1:N、N:M）？通过什么机制实现（外键 / 嵌入 / 引用路径）？
- **访问路径**：常见查询/读取模式是什么？存储层如何高效支持（数据库索引 / 目录结构 / 文件命名规则）？
- **并发控制**：哪些实体存在并发写冲突？用什么策略（悲观锁 / 乐观锁 / 文件锁）？冲突如何恢复？
- **数据迁移**：属性新增/删除/类型变更时，已有数据如何兼容？

> ER 图（Mermaid erDiagram）是存储无关的，适用于所有存储形式。

**3b. 缓存设计**（如适用）：
- 哪些数据需要缓存？为什么（读多写少？热点？计算昂贵？）
- 缓存 Key 格式（如 `order:detail:{orderId}`）
- 过期策略（TTL、LRU、主动失效）和过期时间
- 更新/失效时机（写后失效还是写后更新？）
- 缓存穿透/击穿/雪崩的防范（如布隆过滤器、互斥锁、随机 TTL）

**3c. 消息/事件设计**（如涉及异步）：
- Topic / Queue 名称
- 消息体结构（JSON Schema 或字段说明）
- 生产者逻辑（何时发送、发送失败处理）
- 消费者逻辑（消费成功/失败的后续动作）
- 重试、死信队列、幂等消费机制

**常见遗漏点**：
- 只列了实体和属性，没有定义实体关系
- 只写了缓存 Key，没有过期策略和失效时机
- 涉及异步但没有消息体结构定义
- 没有考虑缓存穿透/击穿/雪崩
- 以存储形式特殊（JSON 文件、文档数据库）为由跳过整个数据设计章节

**质量检查**：接手者只看这一节就能完成数据层的创建（无论存储形式是表、文件还是集合），并正确配置缓存和消息通道。

### 4. 接口设计

**目的**：让上下游能基于接口定义并行开发，减少集成阶段的沟通成本。

**必须覆盖**：

**4a. 对外接口**（REST API、RPC 或消息通道）：
- 协议类型（HTTP/RPC/WebSocket）
- 路径和方法（如 `POST /api/v1/order/calculate`）
- 请求结构（Header、Path Param、Query Param、Body）
- 响应结构（成功和失败两种格式）
- **错误码清单与含义（至少覆盖：参数校验失败、资源不存在、权限不足、服务不可用、未知错误）**
- 鉴权方式（Token、签名、OAuth）

**4b. 内部接口**：
- 各层之间或模块间的方法签名
- 参数类型和含义
- 返回值类型和含义
- 异常约定（抛出什么异常、调用方如何处理）

**4c. 第三方接口**（如调用外部服务）：
- 依赖的服务名称和接口
- 超时时间（连接超时、读取超时）
- 重试策略（重试次数、退避算法、可重试的错误类型）
- 降级策略（调用失败时的兜底行为）
- 熔断策略（熔断阈值、半开探测、恢复条件）

**常见遗漏点**：
- 只写了成功响应结构，没有失败响应和错误码清单——这会导致上下游联调时反复沟通
- 内部接口只有方法名没有异常约定
- 第三方接口没有超时和重试策略，默认值上线后才发现不合理

**质量检查**：前端/下游只看这一节就能写出完整的调用代码和错误处理逻辑。

### 5. 关键算法 / 业务规则实现

**目的**：让有实现歧义的规则在编码前就有明确的逻辑表达。

**"关键规则"的判断很简单**：spec-delta 中出现了「根据…计算」「按照…规则」「当…时则」「满足…条件时」「如果…那么…」等需要"算"或"判"的表述，它就是关键规则，必须纳入 §5 展开。

如果不确定要不要写——就写。宁可多写一条伪代码，也不要让两个开发者对同一条规则做出两种实现。

**必须覆盖**：
- 规则描述：自然语言 + 至少 2 个具体示例（含正常值和边界值）
- 伪代码或逻辑表达式
- 方案对比（如有多种实现路径）
- 推荐方案与理由

**如果确实无关键规则**：写「本功能无关键算法 / 业务规则」并引用 spec-delta 原文说明判断依据。

**常见遗漏点**：
- 规则描述只有一句话，缺少具体示例——比如「根据用户等级给予折扣」没写各等级的具体折扣
- 以为规则简单就不用写——一条 if/else 写出来也比两个人各自理解强
- 以「本功能无关键算法」跳过，但未引用 spec-delta 原文佐证

**质量检查**：另一个开发者只看这一节，就能写出正确的实现，且边界行为一致。

### 6. 异常处理与边界条件

**目的**：提前想清楚「出错了怎么办」，避免线上故障时临时救火。

**必须覆盖**：
- **异常场景列表**：至少覆盖数据非法（参数校验失败、数据格式错误）、依赖失败（数据库宕机、缓存不可用、第三方超时）、并发冲突（乐观锁失败、死锁）、业务异常（库存不足、余额不足、状态不允许）
- **处理策略**：每种异常是重试、降级、拒绝还是补偿？重试需要幂等支持吗？
- 日志与告警：哪些异常需要 ERROR 级别日志？哪些需要触发告警？
- **边界条件**：分页极限值、空列表、null 参数、重复提交、超大数据量、时间为零值/负值
- **幂等性设计**：哪些操作需要幂等？用什么机制实现（唯一键、Token、状态机）？
- 并发冲突处理：乐观锁 vs 悲观锁，选型和理由

**常见遗漏点**：
- 只列了「库存不足」这种业务异常，没考虑「数据库连接超时」这种基础设施异常
- 提了幂等但没有说具体实现机制
- 边界条件只考虑了正常范围，没考虑极端值（0、负数、超长字符串、极限并发）

**质量检查**：运维只看这一节就知道哪些异常需要告警、哪些可以自动恢复。

### 7. 非功能性设计

**目的**：确保功能上线后不只是「能用」，而是「好用、安全、可维护」。

**必须覆盖**：
- **性能**：预估 QPS/TPS，关键路径的响应时间目标（P50/P99），是否需异步化或批量处理
- **安全**：权限校验点（哪些接口需要什么权限）、敏感数据脱敏（日志/响应中哪些字段需要脱敏）、防注入（SQL 注入、XSS）、接口限流
- **监控与日志**：关键步骤的日志输出点（如"生成订单开始"、"生成订单成功"）、业务指标埋点（如"订单创建成功"计数器）、链路追踪 ID 的传递
- **配置**：提取到配置中心的开关、阈值、参数（如"是否开启新算法"、"优惠券最大叠加数"）
- 限流/降级/批处理的具体参数

**常见遗漏点**：
- 性能预估写「无特殊要求」但实际可能有热点
- 安全只写「需要登录」但没有具体权限校验点
- 监控只写了「记录日志」但没有具体埋点位置和指标名称

**质量检查**：SRE 只看这一节就能配置告警规则和监控面板。

### 8. 测试与可验证性建议

**目的**：为测试阶段提供明确的覆盖目标，不等到测试时才发现设计遗漏。

**必须覆盖**：
- **正常场景**：至少 2-3 个典型 happy path
- **异常场景**：至少覆盖第 6 节中的 3 个关键异常
- **边界场景**：至少覆盖空值、极限值、并发
- 测试数据构造方式（如何准备测试数据、如何清理）
- Mock 点（哪些外部依赖需要 Mock，如第三方服务、数据库、缓存）

**常见遗漏点**：
- 只写了正常场景，没写异常和边界
- Mock 点模糊——「Mock 数据库」太粗糙，应该精确到「Mock UserDAO.findById」

**质量检查**：测试工程师只看这一节就能写出完整的测试用例清单。

## 系统架构 / ADR 前置

```text
IF {PROJECT_ROOT}/specflow/architecture.md 不存在 且本次变更需要生成代码或确定实现结构:
  暂停并进入 System Architecture / ADR 阶段
  不生成 design 草案
  不进入 Tasks 或 Apply
ELSE IF 本次变更涉及 ADR 适用范围中的长期决策且对应 ADR 缺失:
  暂停并进入 System Architecture / ADR 阶段
  不生成 design 草案
  不进入 Tasks 或 Apply
ELSE:
  读取 architecture.md 中的两张 Mermaid UML 和 ADR 作为本次 Design 的约束输入
```

"需要生成代码或确定实现结构"包括新增源码、选择技术栈、创建目录结构、定义持久化格式、定义对外接口、确定跨模块数据流、引入依赖或基础设施。

## 讨论循环

```text
读取当前 change 目录中的 proposal、spec-delta，以及主 spec、architecture、ADR、源码和配置事实
识别本次实现问题
识别是否需要新增或调整系统边界图、系统架构图或 ADR

按 8 个设计维度的顺序逐项展开技术实现方案：
  FOR EACH 维度：
    根据"设计维度与质量标准"中的必须覆盖项展开讨论
    检查是否遗漏该维度的常见遗漏点
    对照质量检查项验证完整性
    IF 该维度发现的新增/修改会影响系统边界图、系统架构图或 ADR：
      暂停并进入 System Architecture / ADR 阶段
    ELSE IF 该维度通过质量检查：
      将确认结果写入 design.md 对应章节
       进入下一维度

检查所有 8 个维度是否完整，特别确认"数据设计"和"异常处理与边界条件"未被弱化
IF 无开放问题:
  进入 Tasks 阶段
```

## 架构影响判定

```text
IF 技术内容只是本次实现的内部组织方式:
  写入"模块与分层职责""核心流程 UML""数据设计"或"接口设计"
  architecture_update = no，除非存在其他系统图影响
ELSE IF 技术内容改变系统外部关系、外部系统、用户/角色、信任边界、主要模块/服务/容器、数据存储、关键集成或部署边界:
  写入"架构影响"
  暂停并进入 System Architecture / ADR 阶段
```

内部 service、class、function、helper、目录名或文件名本身不自动构成架构影响。只有当它改变系统边界图或系统架构图时，才升级为架构影响。

## ADR 影响判定

```text
IF 决策属于技术栈选型、核心架构约束、可观测性、数据所有权与存储策略、认证/安全方案、关键集成方案或 AI 协作禁止项:
  在"System Architecture / ADR 影响"中标记 adr_needed = yes
  暂停并进入 System Architecture / ADR 阶段
ELSE:
  只记录在 design.md
```

Design 不创建或修改 `specflow/architecture.md` 或 ADR。系统架构和 ADR 的首次创建、后续更新、替代或废弃均在 System Architecture / ADR 阶段完成。AI 的推荐方案、推断、草案或沉默通过均不等同于用户确认。

## 避坑约束

以下规则在讨论循环和输出 design.md 时强制执行：

1. **禁止只画 UML 而忽略数据设计和异常处理**：数据设计和异常处理是线上故障的两大重灾区。如果在讨论循环中数据设计（维度 3）或异常处理（维度 6）的内容明显弱于 UML（维度 2），暂停并补齐。
2. **时序图必须真实反映调用层次**：Controller、Service、DAO 必须各自独立生命线。禁止把多个层级合并到一条生命线中（如一条「后端」生命线替代所有内部组件）。
3. **接口设计必须包含完整错误码清单和含义**：设计 §4 的对外接口必须列出至少 5 个错误码（参数校验失败、资源不存在、权限不足、服务不可用、未知错误）及其含义。缺少错误码清单的设计视为不完整。
4. **数据设计不得因存储形式特殊而跳过或弱化**：设计 §3 必须覆盖实体、属性、关系、访问路径、并发控制和数据迁移，无论存储形式是表、文件还是集合。如果只覆盖了数据模型而跳过了缓存和消息/事件（当适用时），暂停并补齐。
5. **讨论循环不可跳过任何维度**：即使某个维度看起来简单或无内容，也必须明确写「本功能不涉及 X」并说明理由，不能静默跳过。
6. **§5 不得以"无关键算法"为由跳过而未引用 spec-delta 原文佐证**：如果 design 中 §5 为空，必须确认 spec-delta 中不存在需要「算」或「判」的表述。

## 阻断

```text

  暂停并在 {PROJECT_ROOT}/specflow/changes/<change-id>/ 创建 design.md
ELSE IF 缺少 architecture.md，且本次变更需要生成代码或确定实现结构:
  暂停并进入 System Architecture / ADR 阶段
ELSE IF 本次变更涉及 ADR 适用范围中的长期决策且对应 ADR 缺失:
  暂停并进入 System Architecture / ADR 阶段
ELSE IF design 与 spec-delta 描述的行为冲突:
  暂停并修正冲突
ELSE IF design 引入系统图变更或 ADR 适用范围内的长期决策:
  暂停并进入 System Architecture / ADR 阶段
ELSE IF design 标记 architecture_update = yes 但没有说明同步内容:

ELSE IF design 标记 adr_needed = yes 但没有 ADR 候选:

```
