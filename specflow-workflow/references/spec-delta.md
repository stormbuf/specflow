# Spec Delta Stage

本文件定义规约变更阶段。Spec Delta 描述本次对系统可观察行为的增量变化，归档后合并回主 spec。

路径锚定：本文件中的 `specflow/`、源码、测试和项目文档路径均相对于 `{PROJECT_ROOT}/`；不得在 `{SKILL_DIR}/` 下查找项目规约或变更目录。

## 目标

- 描述新增、修改、删除或重命名的系统行为。
- 每个需求至少包含一个可验证场景。
- 不写实现方案、文件名、类名、算法步骤。
可观察行为的定义沿用 Proposal 阶段的标准：UI 行为、API 响应、CLI 参数、错误语义、状态变化、数据保留策略、同步或一致性语义。

## 输入

- `specflow/changes/<change-id>/proposal.md`
- 已有 `specflow/specs/<capability>.md`，如果修改既有能力。

## 事实边界

Spec Delta 只能基于以下事实来源生成：

- `proposal.md` 中已经确认的行为。
- 已有 `specflow/specs/<capability>.md` 中的长期规约。
- 已有源码、测试或项目文档中可验证的当前系统事实。

不得静默引入上述来源之外的新业务约束。缺失的行为约束必须写入开放问题，并使用苏格拉底追问询问用户。

```text
IF 需求文本需要补充业务规则才能形成可验证场景:
  先检查 proposal、主 spec、源码、测试或项目文档是否已有事实依据
ELSE IF 没有事实依据:
  将缺失约束写入开放问题
  IF 缺失约束影响 Given / When / Then 结构（无法形成可验证需求）:
    标记为阻断性开放问题
    使用苏格拉底追问询问用户并暂停
  ELSE:
    标记为记录性开放问题
    带入后续阶段，不阻断
ELSE:
  基于已确认事实写入 spec-delta
```

## 苏格拉底追问

追问目标是让用户确认业务事实，而不是让 AI 替用户做决定。

```text
IF 缺失约束影响需求是否可验证:
  先指出当前无法确定的事实
  再说明该事实为什么影响可观察行为或验收场景
  最后提出 1-3 个聚焦问题
ELSE:
  不追问，继续规范化已确认行为
```

问题应遵守：

- 不诱导用户接受 AI 的默认方案。
- 不一次性抛出大量选项。
- 优先询问能决定场景 Given / When / Then 的事实。
- 如果用户回答仍然模糊，继续追问边界条件或反例。

示例：

```text
当前 proposal 只说明“用户可以导出订单”，但没有说明导出对象和格式。
这会影响权限场景和返回结果，因此无法直接写入可验证 spec。

1. 导出范围是“当前用户自己的订单”，还是还包括管理员导出他人订单？
2. 导出格式是否固定为 CSV？
3. 未登录用户请求导出时，系统应拒绝还是引导登录？
```

连续 2 轮追问后用户仍无法给出确定性回答：降级为记录性开放问题，标注「待用户确认」并继续。

## 开放问题分类

阻断性开放问题：答案会改变场景 Given / When / Then 结构，无法形成可验证需求。
  → 必须通过苏格拉底追问解决。
  → 连续 2 轮追问后仍无结论，降级为记录性，标注「待用户确认」。

记录性开放问题：不改变可验证性，但影响具体行为边界
  （如「未登录用户行为未定义」、「导出格式未确定」）。
  → 不阻断 spec-delta 完成，带入后续阶段。

## 输出

- `specflow/changes/<change-id>/spec-delta.md`

## 格式

使用以下章节；没有内容的章节可省略：

- `## 目标主规约`
- `## 新增需求`
- `## 修改需求`
- `## 删除需求`
- `## 重命名需求`
- `## 开放问题`

每个需求使用：

```markdown
### 需求：<名称>

<行为描述，使用必须/禁止等规范性词汇>

#### 场景：<场景名称>

- Given <前置条件>
- When <动作>
- Then <结果>
```

> Given / When / Then 必须使用用户或外部系统可观察的语言。
> 不得包含内部方法名、数据库表名、缓存 key、类名或代码路径。

## 修改既有需求

```text
IF 修改 specflow/specs/<capability>.md 中的既有需求:
  复制完整需求块到修改需求
  修改完整块以反映新行为
ELSE IF 只是新增关注点且不改变既有行为:
  使用新增需求
```

## 删除和重命名

删除需求必须包含：

- 原因
- 迁移方案，若不适用则写 none

重命名需求必须包含：

- 原名称
- 新名称

## 阻断

```text
IF proposal 的“是否需要规约”为 yes 但没有当前 change 目录中的 spec-delta.md:
  暂停并在 {PROJECT_ROOT}/specflow/changes/<change-id>/ 创建 spec-delta.md
ELSE IF spec-delta 没有目标主规约:
  暂停并补齐目标 {PROJECT_ROOT}/specflow/specs/<capability>.md
ELSE IF 任一需求没有场景:
   暂停并补齐场景
ELSE IF 修改需求在原主规约中不存在对应需求块:
   暂停，确认需求名称是否正确或是否应为新增需求
ELSE IF 存在阻断性开放问题:
   暂停，解决阻断性开放问题
ELSE IF 场景无法作为测试或验收来源:
  重写场景
ELSE IF spec-delta 包含没有事实来源的新业务约束且影响 Given / When / Then 结构:
   暂停，移入阻断性开放问题并使用苏格拉底追问询问用户
```

## 完成条件

- `{PROJECT_ROOT}/specflow/changes/<change-id>/spec-delta.md` 存在。
- 所有需求均描述可观察行为。
- 每个需求至少一个场景。
- 修改既有需求时包含完整更新后的需求块。
- 无未标明事实来源的新业务约束；缺失约束已进入开放问题。
- 阻断性开放问题已解决；记录性开放问题可带入后续阶段。
- 删除需求包含原因和迁移方案；重命名需求包含原名称和新名称。

## 完整示例

以下是以「订单导出」变更为例的 spec-delta：

### 目标主规约

- specflow/specs/order.md

### 新增需求

### 需求：订单导出

系统必须允许用户导出自己的订单列表。

#### 场景：已登录用户导出近 30 天订单

- Given 用户已登录且近 30 天有 3 笔订单
- When 用户在订单列表页点击"导出"按钮
- Then 系统返回一个 CSV 文件，包含订单号、商品名称、数量、金额、下单时间
- 且文件名格式为 `orders_<YYYY-MM-DD>_<HHmm>.csv`

### 修改需求

### 需求：订单列表查询

（以下为从 specflow/specs/order.md 复制的完整块，已更新以反映导出支持）

系统必须支持按时间范围、状态、关键词查询订单。

> 原需求仅支持按状态筛选，本次新增时间范围和关键词维度。

#### 场景：按时间范围查询

- Given 用户已登录且存在 2026-05-01 至 2026-05-31 的订单
- When 用户选择时间范围 2026-05-01 至 2026-05-31 并点击查询
- Then 系统返回该时间范围内的所有订单

#### 场景：按关键词查询

- Given 用户已登录且存在包含"显示器"的订单
- When 用户在搜索框输入"显示器"并点击查询
- Then 系统返回商品名称包含"显示器"的订单

### 删除需求

### 需求：订单导出（旧版 PDF 导出）

- 原因：旧版 PDF 导出功能仅支持单订单导出，且依赖已废弃的 PDF 渲染库。
- 迁移方案：用户迁移至新版 CSV 导出（见新增需求「订单导出」），批量导出能力由 CSV 替代。

### 开放问题

1. **[记录性]** 管理员是否需要导出其他用户的订单？当前仅覆盖当前用户导出自身订单。
2. **[记录性]** 导出文件是否需要包含退款/售后订单？当前未定义。
