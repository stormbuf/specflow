## 禁止模式

（由团队填写：列出代码库中绝对不允许的模式。每条配"为什么不"的说明。例如：

- 禁止使用 `any` 类型绕过类型检查
- 禁止直接操作 DOM（`document.querySelector`），用 ref 替代
- 禁止在组件中硬编码 API 地址
- 禁止使用 `// @ts-ignore` / `// @ts-expect-error` 而无注释说明
- 禁止内联魔法数字与字符串，必须提取为常量
- 禁止在 render 中执行副作用（如直接 setState）
）

## 必需模式

（由团队填写：列出代码库中必须遵循的模式。例如：

- 所有组件必须有明确返回类型
- 所有 API 响必须有类型定义
- 所有用户输入必须有校验
- 所有列表渲染必须有稳定的 key
- 所有异步操作必须处理 loading 与 error 态
）

## 测试要求

（由团队填写：定义测试的最低要求。包括：

- 测试框架（Vitest / Jest 等）
- 组件测试库（Testing Library / Enzyme 等）
- 覆盖率门槛
- 哪些必须测试（hook / 工具函数 / 关键交互组件）
- E2E 测试的范围与框架（Playwright / Cypress）
- 测试命名规则
）

## 代码审查检查清单

（由团队填写：定义 reviewer 在 review 时必须逐项确认的清单。例如：

- [ ] 无禁止模式
- [ ] 类型完整，无 any
- [ ] 有对应测试且通过
- [ ] 组件处理了 loading / error / empty 三态
- [ ] 列表有稳定 key
- [ ] 无直接 DOM 操作
- [ ] 无内联魔法数字
- [ ] 可访问性达标
- [ ] 无明显性能问题（大列表未虚拟化、无意义重渲染）
）

## 常见错误

（由团队填写：列出团队在质量把控上犯过的错误，例如：

- 测试只渲染不断言，覆盖率虚高
- 用 `any` 快速通过类型检查，运行时崩溃
- 列表用 index 作 key，导致渲染错乱
- review 不看可访问性，上线后键盘用户无法操作
- bundle 无人监控，体积悄悄膨胀
）
