## 如何填写这些规范

这些骨架文件存在的目的是记录团队**真实在做**的约定，而非理想中的最佳实践。填写时遵循以下原则：

1. **记录实际约定，而非理想** — 写下当前代码库里真正遵循的规则。如果"理想"和"实际"有差距，先记实际，再标注差距与改进方向。空喊最佳实践没有约束力。
2. **包含代码示例** — 每条规则至少配一个"推荐"与"不推荐"的代码片段。AI 与新成员需要可复制的样板，而不是抽象描述。
3. **列出禁止模式** — 明确写出"不要这样做"清单。禁止项比推荐项更有约束力，能让 AI 在生成代码时主动规避。
4. **添加团队犯过的常见错误** — 把历史踩过的坑写进"常见错误"章节。这些经验是花钱买来的，不记录就会重复犯。

## 编码前检查清单

在编写或修改 API 端点前，逐项确认：

- [ ] 我知道 URL 命名约定（复数 / 单数、嵌套层级）
- [ ] 我知道 HTTP 方法的语义约定（GET / POST / PUT / PATCH / DELETE）
- [ ] 我知道请求 / 响应的 JSON 字段命名风格（camelCase / snake_case）
- [ ] 我知道分页、过滤、排序的统一参数约定
- [ ] 我知道错误响应的统一格式与错误码体系
- [ ] 我知道 API 版本管理策略与向后兼容要求

## 语言要求

- 所有规范文档使用中文撰写，技术术语（如 REST、endpoint、payload、pagination）保留英文原词。
- 代码示例中的标识符、注释可使用英文，但解释性文字使用中文。
- 规范描述应面向"让 AI 能直接执行"的粒度，避免模糊表述（如"API 要规范"应改写为具体规则）。
