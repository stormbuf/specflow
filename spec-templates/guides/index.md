# 思维指南

> **核心原则**：30 分钟思考节省 3 小时调试。

---

## 为什么需要思维指南

**大多数 bug 和技术债务来自"没想到"，而非技能不足。**

你完全有能力写出正确的代码，但如果你在动手前没有想到：

- 这个逻辑是不是已经存在了？
- 数据跨层传递时格式是否一致？
- 这段代码在另一个平台上会怎样？
- 修改这个值会不会影响其他地方？

那么 bug 就会在你以为"写完了"的时候悄然埋下。思维指南的作用就是在你动手之前，强迫你花几分钟想过这些"没想到"的问题。

## 思维触发器检查清单

当你遇到以下情况时，先读对应的指南：

| 触发器 | 该读的指南 |
|--------|-----------|
| 我要写一个新函数 / 新组件 | 代码复用思维指南 |

## 修改前关键规则：改任何值之前，先搜索！

> **这是最重要的一条规则。**

在你修改任何一个值（配置项、常量、路径、字符串、枚举值）之前，**先全局搜索这个值的所有出现位置**。

为什么？因为：

- 一个值可能被硬编码在多个文件中
- 修改一处而遗漏其他处，是最常见的"改了 A 坏了 B"的原因
- 枚举值的新增可能导致 `if/elif/else` 分支静默 fallthrough

### 搜索方法

```bash
# 搜索某个值的所有出现位置
grep -r "你要改的值" .

# 搜索函数名 / 变量名
grep -r "functionName" .

# 只搜索特定文件类型
grep -r "keyword" --include="*.go" --include="*.yaml"
```

### 修改前的检查清单

- [ ] 我已经搜索了要修改的值的所有出现位置
- [ ] 我知道每一处出现的上下文（是否需要同步修改）
- [ ] 如果这个值是枚举 / Literal 类型，我检查了所有 switch / if-else 分支
- [ ] 如果这个值是路径，我检查了跨平台兼容性

## 如何使用这些指南

1. **动手前读**：在开始写代码前，根据触发器检查清单找到对应的指南
2. **边写边对照**：把检查清单当作你的草稿纸，逐项确认
3. **提交前复查**：提交代码前，再过一遍检查清单

这些指南不是教条，而是**经验教训的浓缩**。每一条规则背后都可能是一个真实存在的 bug。

---
