## 日志级别

（由团队填写：定义每个日志级别的使用场景。例如：

- `error` — 系统错误，需要人工介入
- `warn` — 可预期的异常情况，如限流触发、降级
- `info` — 关键业务事件，如订单创建、支付完成
- `debug` — 开发调试用，生产关闭
- `trace` — 极细粒度链路，仅本地调试
）

## 结构化日志

（由团队填写：定义结构化日志的字段规范。例如：

```json
{
  "timestamp": "2026-07-01T10:00:00Z",
  "level": "info",
  "trace_id": "abc-123",
  "module": "order",
  "event": "order_created",
  "order_id": "456",
  "user_id": "789"
}
```

包括：必填字段、字段命名规则、trace_id 如何注入与传播。）

## 记录内容

（由团队填写：定义应该记录的内容清单。包括：

- 请求入口与出口（method、path、status、耗时）
- 关键业务状态变更
- 外部调用（RPC、第三方 API）的入参与响应摘要
- 异常与错误（含 stack）
- 慢查询与慢操作
）

## 禁止记录内容

（由团队填写：定义绝对不能记录的敏感信息。包括：

- 用户密码（任何形式）
- token / API key / 密钥
- 完整手机号、身份证号、银行卡号（需脱敏，如 `138****1234`）
- 用户完整地址
- 请求 / 响应中的 PII 字段
- 大体积 payload 全文（应截断或摘要）
）

## 常见错误

（由团队填写：列出团队在日志上犯过的错误，例如：

- 把整个请求 body 打进日志，包含用户密码
- 用 `console.log` 而非日志库，生产环境无法统一收集
- 日志级别乱用，info 里塞了 debug 级细节，生产日志爆炸
- 没有 trace_id，线上问题无法串联调用链
- 循环里打日志，单次请求产生上万条日志
）
